# 模型微调与部署

> **一句话先说清楚**：微调不是让模型"变聪明"，而是让模型"变专业"——把一个全科医生训练成你诊所的专科大夫，说话风格、回答格式、领域知识都贴合你的业务。

---

## 第一部分：先想清楚要不要微调（总）

### 微调、RAG、Prompt 工程 —— 三种方案怎么选？

很多人一上来就问"怎么微调"，但更重要的问题是"**是否真的需要微调**"。

把需求扔进这张决策表：

```
你的问题是什么？
│
├── 模型不知道你的私有数据（内部文档、数据库）
│   └── ✅ 用 RAG，不需要微调
│       原因：微调不能注入知识，它改变的是行为模式
│
├── 模型输出格式不对（你要 JSON，它输出自然语言）
│   ├── 先试 Prompt 工程（system prompt + few-shot）
│   └── Prompt 控制不住 → ✅ 微调
│
├── 模型风格不对（太啰嗦、太正式、不符合品牌调性）
│   ├── 先试 Prompt 工程
│   └── 依然不稳定 → ✅ 微调
│
├── 模型不懂你的领域术语（医疗、法律、金融专有名词）
│   ├── 数据量小 → RAG + 领域文档
│   └── 需要模型内化理解 → ✅ 微调
│
├── 推理成本太高（用 GPT-4 太贵，想用小模型替代）
│   └── ✅ 微调小模型（知识蒸馏）
│
└── 模型幻觉太多
    └── ❌ 微调解决不了，考虑 RAG + 事实核查
```

**一句话总结**：微调改变的是**模型的行为方式**，不是**模型知道什么**。

---

### 三种方案的成本对比

| 方案 | 实施成本 | 推理成本 | 效果稳定性 | 适用场景 |
|------|---------|---------|-----------|---------|
| Prompt 工程 | 低（小时级） | 高（每次带大量 prompt） | 中（输出可能漂移） | 快速验证、通用任务 |
| RAG | 中（天级） | 中 | 高（可追溯） | 私有知识库、实时数据 |
| 微调 | 高（天～周级） | 低（小模型快） | 高（行为固化） | 格式固定、高频调用、成本敏感 |

**微调什么时候经济上划算？**

```
每天调用次数 × (大模型成本 - 小模型成本) > 微调一次的成本
              ↑
      当这个差值足够大时，微调才合理
```

如果每天只调用几百次，微调的收益很难覆盖成本。

## 第二部分：微调的原理（分）

### 2.1 全参数微调 vs 参数高效微调

**全参数微调（Full Fine-tuning）**

把模型所有参数都更新。

```
Llama-3-8B = 80 亿个参数
全参数微调 = 更新全部 80 亿个参数
显存需求 ≈ 模型权重(16GB) + 梯度(16GB) + 优化器状态(32GB) ≈ 64GB+
→ 需要多张 A100，成本极高
```

大部分人用不起，也没必要。

**参数高效微调（PEFT, Parameter-Efficient Fine-Tuning）**

冻结绝大多数参数，只训练极少数新增参数。

```
全参数微调：更新 8,000,000,000 个参数
LoRA 微调：只更新       4,000,000 个参数（0.05%）
效果差距：微小
成本差距：巨大
```

PEFT 是目前的主流方法，你绝大多数场景用这个。

---

### 2.2 LoRA 原理——最重要的概念

**LoRA（Low-Rank Adaptation）** 是当前最流行的 PEFT 方法。

**直觉理解：**

想象你要修改一幅 10000×10000 的画（模型权重矩阵 W）。

- 全参数微调：把整幅画重新画一遍，需要 1 亿个像素的存储
- LoRA：不动原画，在旁边画两个小图（100×10 和 10×100），最终效果叠加上去

```
原始权重矩阵 W（冻结，训练时不更新）
                    +
低秩矩阵 A（随机初始化，训练更新）
         ×
低秩矩阵 B（初始化为零，训练更新）

实际效果：W' = W + A × B
```

**为什么叫"低秩"？**

```
原矩阵 W 的形状：d × d（例如 4096 × 4096 = 1600万参数）

LoRA 的 A 矩阵：d × r（例如 4096 × 16 = 6.5万参数）
LoRA 的 B 矩阵：r × d（例如 16 × 4096 = 6.5万参数）

其中 r（rank）= 16，远小于 d = 4096
A × B 的结果形状 = d × d，和 W 一样
但参数量从 1600万 → 13万，压缩了 120 倍

r 是最核心的超参数：
  r 越大 → 拟合能力越强，参数越多
  r 越小 → 参数越少，可能欠拟合
  通常设置 r = 8 ~ 64
```

**LoRA 加在哪里？**

```
Transformer 结构
│
├── Attention 层
│   ├── Q 矩阵（Query）  ← LoRA 加这里
│   ├── K 矩阵（Key）    ← LoRA 加这里
│   ├── V 矩阵（Value）  ← LoRA 加这里
│   └── O 矩阵（Output） ← LoRA 加这里
│
└── FFN 层（通常不加，也可以加）
```

---

### 2.3 QLoRA——消费级 GPU 上的微调

QLoRA = **量化（Quantization）** + **LoRA**

**量化**：把模型权重从 16-bit 压缩到 4-bit，显存减少 4 倍，精度损失可接受。

```
Llama-3-8B 不同精度的显存占用：
  FP16（16位浮点）：~16 GB
  INT8（8位整数）：~8 GB
  INT4（4位整数）：~4 GB  ← QLoRA 使用这个加载基础模型
  
QLoRA 完整流程：
  基础模型（INT4 量化加载）≈ 4 GB  → 大幅减少显存
         +
  LoRA 适配器（FP16 训练）≈ 0.5 GB → 训练精度不损失
         =
  8B 模型在单张 8-12 GB 消费级显卡上可以微调！
```

**显卡门槛对照：**

| 模型规模 | 全参数微调 | QLoRA 微调 |
|---------|-----------|-----------|
| 1-3B | A10（24G）× 2 | RTX 3060（12G）|
| 7-8B | A100（80G）× 2 | RTX 3090（24G）|
| 13B | A100（80G）× 4 | RTX 4090（24G）|
| 70B | A100（80G）× 8 | A100（80G）× 2 |

**没有高端 GPU？用云 GPU：**
- Google Colab Pro（A100，约 ¥140/月）
- AutoDL（国内，按小时计费，RTX 4090 约 ¥3/小时）
- Vast.ai（海外，A100 约 $1-2/小时）

## 第三部分：数据准备（最被忽视，最影响效果）

> **数据质量 > 数据数量 > 模型规模**
> 500 条精标数据，比 10000 条噪声数据效果好得多。

```json
    ]
  }

```

### 3.2 数据质量标准

**一条好数据应该满足：**

```
✅ instruction 清晰，没有歧义
✅ output 是你期望模型输出的"理想答案"
✅ output 风格、长度、格式一致（不要这条很简短，那条很详细）
✅ 覆盖边界情况（输入为空、输入有误时如何处理）
✅ 没有错误信息（检查事实准确性）

❌ instruction 和 output 对应关系不准确
❌ output 是"差不多能用"而不是"最佳答案"
❌ 数据来源混杂（风格不一致会让模型困惑）
```

**数量建议（经验值）：**

| 任务类型 | 最少数据量 | 推荐数据量 |
|---------|-----------|-----------|
| 格式转换（JSON/Markdown） | 100-300 条 | 500 条 |
| 风格改写 | 300-500 条 | 1000 条 |
| 领域问答 | 500-1000 条 | 2000 条 |
| 代码生成 | 1000-2000 条 | 5000 条 |
| 复杂推理 | 5000+ 条 | 10000+ 条 |

```python
import anthropic
import json

client = anthropic.Anthropic()

def generate_training_data(task_description: str, num_examples: int = 50):
    """用 Claude 生成训练数据"""

    prompt = f"""
你是一个数据标注专家。请为以下任务生成 {num_examples} 条高质量训练数据。

任务描述：{task_description}

要求：
1. instruction 要多样化，覆盖不同输入场景
2. output 必须是该任务的理想答案
3. 每条数据都要完整、准确
4. 输出严格的 JSON 数组格式

输出格式：
[
  {{"instruction": "...", "input": "...", "output": "..."}},
  ...
]
"""
    response = client.messages.create(
        model="claude-opus-4-5",   # 用最强模型生成数据
        max_tokens=8192,
        messages=[{"role": "user", "content": prompt}]
    )

    raw = response.content[0].text
    # 提取 JSON 部分
    start = raw.find("[")
    end = raw.rfind("]") + 1
    return json.loads(raw[start:end])


# 使用示例
data = generate_training_data(
    task_description="将用户投诉文本分类为：产品质量、物流延迟、客服态度、退款问题、其他，输出 JSON 格式",
    num_examples=100
)

# 保存
with open("training_data.json", "w", encoding="utf-8") as f:
    json.dump(data, f, ensure_ascii=False, indent=2)

print(f"生成了 {len(data)} 条训练数据")
```

### 3.4 数据检查脚本

**微调前必做的数据检查：**

```python
import json

def check_dataset(file_path: str):
    with open(file_path, "r", encoding="utf-8") as f:
        data = json.load(f)

    issues = []

    for i, item in enumerate(data):
        # 检查必填字段
        if "instruction" not in item:
            issues.append(f"第 {i+1} 条：缺少 instruction 字段")

        if "output" not in item:
            issues.append(f"第 {i+1} 条：缺少 output 字段")
            continue

        # 检查长度
        if len(item.get("output", "")) < 10:
            issues.append(f"第 {i+1} 条：output 过短（{len(item['output'])} 字符）")

        if len(item.get("output", "")) > 4000:
            issues.append(f"第 {i+1} 条：output 过长（{len(item['output'])} 字符），可能超出模型输出限制")

        # 检查空值
        if not item.get("instruction", "").strip():
            issues.append(f"第 {i+1} 条：instruction 为空")

    # 输出统计
    outputs = [item.get("output", "") for item in data]
    avg_len = sum(len(o) for o in outputs) / len(outputs)

    print(f"✅ 数据集统计：")
    print(f"   总条数：{len(data)}")
    print(f"   output 平均长度：{avg_len:.0f} 字符")
    print(f"   output 最短：{min(len(o) for o in outputs)} 字符")
    print(f"   output 最长：{max(len(o) for o in outputs)} 字符")

    if issues:
        print(f"\n⚠️  发现 {len(issues)} 个问题：")
        for issue in issues[:10]:   # 只显示前 10 个
            print(f"   {issue}")
    else:
        print(f"\n✅ 未发现问题，数据集可以使用")

check_dataset("training_data.json")
```

### 4.2 完整微调脚本（QLoRA + SFTTrainer）

```python
"""
完整 QLoRA 微调脚本
基础模型：Qwen2.5-7B（推荐中文场景）
显卡要求：单张 16GB+ 显存（RTX 3090 / 4090 / A10）
"""

import torch
from datasets import load_dataset, Dataset
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    BitsAndBytesConfig,
    TrainingArguments,
)
from peft import LoraConfig, get_peft_model, TaskType
from trl import SFTTrainer
import json

# ─── 配置区（修改这里）────────────────────────────────────────

MODEL_NAME = "Qwen/Qwen2.5-7B-Instruct"  # 基础模型
DATA_PATH = "training_data.json"           # 训练数据
OUTPUT_DIR = "./qwen2.5-finetuned"        # 输出目录

# ─── 步骤一：量化配置（QLoRA 的关键）────────────────────────

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                     # 4bit 量化加载模型
    bnb_4bit_quant_type="nf4",            # NF4 量化类型（精度最好）
    bnb_4bit_compute_dtype=torch.bfloat16, # 计算时用 BF16（A100+）
    bnb_4bit_use_double_quant=True,        # 双重量化，进一步节省显存
)

# ─── 步骤二：加载模型和 Tokenizer ───────────────────────────

print("正在加载模型（第一次运行会下载，需要等待）...")
model = AutoModelForCausalLM.from_pretrained(
    MODEL_NAME,
    quantization_config=bnb_config,
    device_map="auto",                     # 自动分配 GPU/CPU
    trust_remote_code=True,
)
model.config.use_cache = False             # 训练时不需要 KV Cache
model.config.pretraining_tp = 1

tokenizer = AutoTokenizer.from_pretrained(MODEL_NAME, trust_remote_code=True)
tokenizer.pad_token = tokenizer.eos_token  # 设置 padding token
tokenizer.padding_side = "right"           # 右侧 padding

# ─── 步骤三：LoRA 配置 ──────────────────────────────────────

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                                  # LoRA rank（秩）
    lora_alpha=32,                         # 缩放因子（通常是 r 的 2 倍）
    lora_dropout=0.05,                     # Dropout 防止过拟合
    target_modules=[                       # 要加 LoRA 的模块
        "q_proj", "k_proj", "v_proj",     # Attention 层
        "o_proj",
        "gate_proj", "up_proj", "down_proj" # FFN 层（可选）
    ],
    bias="none",
)

model = get_peft_model(model, lora_config)

# 打印可训练参数量
trainable = sum(p.numel() for p in model.parameters() if p.requires_grad)
total = sum(p.numel() for p in model.parameters())
print(f"可训练参数：{trainable:,} / {total:,} ({100 * trainable / total:.2f}%)")
# 输出示例：可训练参数：20,971,520 / 7,721,324,544 (0.27%)

# ─── 步骤四：准备数据集 ──────────────────────────────────────

def format_instruction(sample: dict) -> str:
    """
    将数据格式化为模型期望的对话模板
    不同模型有不同模板，Qwen2.5 使用 ChatML 格式
    """
    instruction = sample["instruction"]
    input_text = sample.get("input", "")
    output = sample["output"]

    if input_text:
        user_content = f"{instruction}\n\n{input_text}"
    else:
        user_content = instruction

    # Qwen2.5 的对话格式
    return (
        f"<|im_start|>system\n你是一个有用的AI助手。<|im_end|>\n"
        f"<|im_start|>user\n{user_content}<|im_end|>\n"
        f"<|im_start|>assistant\n{output}<|im_end|>"
    )


# 加载数据
with open(DATA_PATH, "r", encoding="utf-8") as f:
    raw_data = json.load(f)

dataset = Dataset.from_list(raw_data)

# 格式化为文本
dataset = dataset.map(
    lambda x: {"text": format_instruction(x)},
    remove_columns=dataset.column_names
)

# 划分训练集和验证集（9:1）
split = dataset.train_test_split(test_size=0.1, seed=42)
train_dataset = split["train"]
eval_dataset = split["test"]

print(f"训练集：{len(train_dataset)} 条，验证集：{len(eval_dataset)} 条")

# ─── 步骤五：训练参数 ────────────────────────────────────────

training_args = TrainingArguments(
    output_dir=OUTPUT_DIR,
    num_train_epochs=3,                # 训练轮数（小数据集可以多训几轮）
    per_device_train_batch_size=4,     # 每块 GPU 的 batch size
    per_device_eval_batch_size=4,
    gradient_accumulation_steps=4,    # 梯度累积（等效 batch=16）
    learning_rate=2e-4,                # 学习率（LoRA 通常比全参数大）
    lr_scheduler_type="cosine",        # 余弦退火调度
    warmup_ratio=0.05,                 # 5% 的步数用于 warmup
    fp16=False,                        # A100+ 用 bf16；旧卡用 fp16=True
    bf16=True,
    logging_steps=10,                  # 每 10 步打印一次日志
    eval_strategy="steps",             # 按步数评估
    eval_steps=100,                    # 每 100 步评估一次
    save_strategy="steps",
    save_steps=100,
    save_total_limit=3,                # 最多保留 3 个 checkpoint
    load_best_model_at_end=True,       # 训练结束加载最佳模型
    report_to="none",                  # 不上报到 wandb（改为 "wandb" 可监控）
    dataloader_num_workers=4,
)

# ─── 步骤六：开始训练 ────────────────────────────────────────

trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
    dataset_text_field="text",         # 数据集中文本字段名
    max_seq_length=2048,               # 最大序列长度
    packing=False,                     # 是否把多个短样本打包（提高效率）
)

print("开始训练...")
trainer.train()

# ─── 步骤七：保存 LoRA 权重 ──────────────────────────────────

# 只保存 LoRA 适配器（几十 MB，不是完整模型）
trainer.model.save_pretrained(f"{OUTPUT_DIR}/lora_adapter")
tokenizer.save_pretrained(f"{OUTPUT_DIR}/lora_adapter")

print(f"LoRA 适配器已保存到 {OUTPUT_DIR}/lora_adapter")

# ─── 步骤八：合并权重（可选，部署用）──────────────────────────

# 如果需要独立的完整模型（用于 vLLM 等部署框架），需要合并权重
from peft import PeftModel

print("合并 LoRA 权重到基础模型...")
base_model = AutoModelForCausalLM.from_pretrained(
    MODEL_NAME,
    torch_dtype=torch.bfloat16,
    device_map="auto",
    trust_remote_code=True,
)
model_merged = PeftModel.from_pretrained(base_model, f"{OUTPUT_DIR}/lora_adapter")
model_merged = model_merged.merge_and_unload()   # 合并 LoRA 权重

model_merged.save_pretrained(f"{OUTPUT_DIR}/merged_model")
tokenizer.save_pretrained(f"{OUTPUT_DIR}/merged_model")
print(f"合并模型已保存到 {OUTPUT_DIR}/merged_model")
```

## 第五部分：量化——让模型跑得更快更省

### 5.1 为什么要量化

```
Qwen2.5-7B 合并后的模型大小：
  FP32（32位）：~28 GB  ← 太大，一般不用
  FP16（16位）：~14 GB  ← 标准大小
  INT8（8位）：~7 GB   ← 速度提升，质量略降
  INT4（4位）：~4 GB   ← 速度再提升，质量有损但可接受
  GGUF Q4_K_M：~4.5 GB ← CPU 推理专用格式
```

量化的本质：把浮点数"压缩"成整数，用精度换速度和显存。

---

### 5.2 GGUF 格式——用于 Ollama / llama.cpp（本地部署首选）

```bash
# 安装 llama.cpp（包含转换脚本）
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
pip install -r requirements.txt

# 步骤一：将 HuggingFace 格式转换为 GGUF
python convert_hf_to_gguf.py \
    /path/to/merged_model \
    --outtype f16 \
    --outfile model-f16.gguf

# 步骤二：量化 GGUF（选择量化级别）
./quantize model-f16.gguf model-q4_k_m.gguf Q4_K_M

# 量化级别对比：
# Q4_K_M：推荐！质量和大小的最佳平衡
# Q5_K_M：质量更好，文件更大
# Q8_0：几乎无质量损失，但文件是 Q4 的 2 倍

# 步骤三：验证模型可以运行
./main -m model-q4_k_m.gguf -p "你好" -n 100
```

### 6.3 FastAPI 封装——对外提供标准接口

```python
"""
将 vLLM 封装为 FastAPI 服务，加入鉴权、日志、限流等生产要素
"""

import time
import uuid
from fastapi import FastAPI, HTTPException, Depends, Header
from fastapi.responses import StreamingResponse
from pydantic import BaseModel, Field
from openai import AsyncOpenAI
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

app = FastAPI(title="AI 推理服务", version="1.0.0")

# 连接 vLLM 后端
vllm_client = AsyncOpenAI(
    base_url="http://localhost:8000/v1",
    api_key="not-needed"
)


# ─── 请求/响应模型 ─────────────────────────────────────────

class ChatRequest(BaseModel):
    message: str = Field(..., description="用户消息", min_length=1, max_length=4000)
    temperature: float = Field(default=0.3, ge=0, le=1)
    max_tokens: int = Field(default=512, ge=1, le=2048)
    stream: bool = Field(default=False)

class ChatResponse(BaseModel):
    request_id: str
    content: str
    usage: dict
    latency_ms: float


# ─── 鉴权中间件 ────────────────────────────────────────────

VALID_API_KEYS = {"sk-prod-xxx", "sk-dev-yyy"}  # 实际从数据库或环境变量读取

async def verify_api_key(x_api_key: str = Header(...)):
    if x_api_key not in VALID_API_KEYS:
        raise HTTPException(status_code=401, detail="无效的 API Key")
    return x_api_key


# ─── 主接口 ─────────────────────────────────────────────────

@app.post("/v1/chat", response_model=ChatResponse)
async def chat(
    request: ChatRequest,
    api_key: str = Depends(verify_api_key)
):
    request_id = str(uuid.uuid4())[:8]
    start_time = time.time()

    logger.info(f"[{request_id}] 收到请求，消息长度：{len(request.message)}")

    try:
        response = await vllm_client.chat.completions.create(
            model="my-model",
            messages=[{"role": "user", "content": request.message}],
            temperature=request.temperature,
            max_tokens=request.max_tokens,
        )

        content = response.choices[0].message.content
        latency = (time.time() - start_time) * 1000

        logger.info(
            f"[{request_id}] 完成，耗时 {latency:.0f}ms，"
            f"输入 {response.usage.prompt_tokens} tokens，"
            f"输出 {response.usage.completion_tokens} tokens"
        )

        return ChatResponse(
            request_id=request_id,
            content=content,
            usage={
                "prompt_tokens": response.usage.prompt_tokens,
                "completion_tokens": response.usage.completion_tokens,
            },
            latency_ms=latency
        )

    except Exception as e:
        logger.error(f"[{request_id}] 推理失败：{e}")
        raise HTTPException(status_code=500, detail=f"推理失败：{str(e)}")


@app.post("/v1/chat/stream")
async def chat_stream(
    request: ChatRequest,
    api_key: str = Depends(verify_api_key)
):
    """流式输出接口（SSE）"""

    async def generate():
        async with vllm_client.chat.completions.create(
            model="my-model",
            messages=[{"role": "user", "content": request.message}],
            temperature=request.temperature,
            max_tokens=request.max_tokens,
            stream=True
        ) as stream:
            async for chunk in stream:
                if chunk.choices[0].delta.content:
                    yield f"data: {chunk.choices[0].delta.content}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")


@app.get("/health")
async def health():
    return {"status": "ok", "model": "my-model"}
```

## 第七部分：从头到尾——完整流程速查

### 7.1 标准工作流

```
1. 确认需求
   └── 用决策表判断是否需要微调（第一部分）

2. 准备数据
   ├── 确定数据格式（指令格式 / 对话格式）
   ├── 收集 / 生成数据（500-2000 条）
   └── 运行数据检查脚本

3. 选择基础模型
   ├── 中文场景：Qwen2.5-7B-Instruct（首选）
   ├── 多语言：Llama-3.1-8B-Instruct
   └── 轻量级（<8GB 显存）：Qwen2.5-3B

4. 微调训练
   ├── 使用 QLoRA（消费级 GPU 可用）
   ├── r=16, lora_alpha=32, epochs=3
   └── 监控 eval_loss，防止过拟合

5. 效果评估
   ├── 准备测试集（100条，覆盖典型场景）
   ├── 对比微调前后的输出
   └── 人工打分或自动评估（LLM 作为评判）

6. 量化
   ├── 本地部署 → GGUF Q4_K_M（Ollama 用）
   └── 服务器部署 → AWQ 4bit（vLLM 用）

7. 部署
   ├── 开发测试 → Ollama
   ├── 生产（低并发）→ vLLM + FastAPI
   └── 生产（高并发）→ vLLM 集群 + Nginx 负载均衡
```

---

### 7.2 常见问题速查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| CUDA OOM（显存不足） | batch size 太大 | 减小 `per_device_train_batch_size`，增大 `gradient_accumulation_steps` |
| loss 不下降 | 学习率太小 / 数据有问题 | 尝试 `lr=2e-4`；检查数据格式是否正确 |
| 过拟合（eval_loss 上升）| 训练轮数太多 / 数据太少 | 减少 `num_train_epochs`；增加数据 |
| 输出格式不符合预期 | 训练数据格式不一致 | 统一数据格式，增加同类型示例 |
| 模型输出混乱（乱码）| 对话模板不匹配 | 确认使用了正确的对话模板格式 |
| vLLM 启动报错 | CUDA 版本不匹配 | `pip install vllm==0.4.x`（与 CUDA 版本对应）|

## 总结：微调的本质

```
微调不是魔法，它是有代价的工程决策：

代价：
  ├── 数据标注成本（时间和金钱）
  ├── 训练计算成本（GPU 费用）
  ├── 维护成本（模型版本管理、持续更新数据）
  └── 机会成本（这些时间用来改 Prompt 也许够了）

收益：
  ├── 更稳定的输出格式（减少解析错误）
  ├── 更低的推理延迟（小模型快）
  ├── 更低的推理成本（自部署 vs API）
  └── 数据私有性（不发送给第三方）

什么时候收益 > 代价：
  ✅ 每天高频调用（> 10 万次）
  ✅ 输出格式必须精确（结构化数据）
  ✅ 对响应速度有严格要求
  ✅ 数据不能出境（金融、医疗、政府）
  ✅ Prompt 工程已经达到上限但效果仍然不够好

微调 + RAG + Prompt 工程不是互斥关系：
  最强方案 = 微调（行为对齐）+ RAG（知识注入）+ 精心设计的 Prompt
```
