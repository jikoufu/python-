## 阶段 4：深度学习基础

目标：能用 PyTorch 训练神经网络，理解 Transformer 架构（大模型的基础）。

### 学习内容

**神经网络基础**

- 感知机 → 多层感知机（MLP）
- 激活函数：ReLU、sigmoid、tanh、GELU
- 前向传播：输入 → 逐层计算 → 输出
- 反向传播：链式法则计算梯度
- 优化器：SGD、Adam（自适应学习率）

**PyTorch 核心**

```python
import torch
import torch.nn as nn

class MLP(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.Sequential(
            nn.Linear(784, 256),
            nn.ReLU(),
            nn.Linear(256, 10)
        )

    def forward(self, x):
        return self.layers(x)

model = MLP()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
loss_fn = nn.CrossEntropyLoss()
```

**Transformer 架构（重点）**

- Self-Attention 自注意力：Q、K、V 矩阵
- Scaled Dot-Product Attention：`softmax(QK^T / sqrt(d_k)) * V`
- Multi-Head Attention：多头并行捕捉不同关系
- 位置编码（Positional Encoding）：注入序列位置信息
- Feed-Forward Network：逐位置的两层 MLP
- Layer Normalization 与残差连接
- Encoder-Decoder 结构 vs 仅 Decoder（GPT）

**训练技巧**

- Dropout：随机丢弃神经元，防止过拟合
- Batch Normalization / Layer Normalization
- 学习率调度：预热（warmup）+ 余弦退火
- 梯度裁剪（Gradient Clipping）

## 阶段 5：大语言模型原理

目标：理解 LLM 是怎么工作的，能解释"为什么 GPT 能生成文字"。

### 学习内容

**语言模型的演进**

- N-gram 语言模型：根据前 N 个词预测下一个词
- Word2Vec / GloVe：词向量，捕捉语义关系
- ELMo：上下文相关的词向量
- BERT（2018）：双向 Transformer Encoder，预训练 + 微调范式
- GPT（2018）：单向 Transformer Decoder，自回归生成
- GPT-2/3：规模化带来涌现能力
- InstructGPT：RLHF 对齐人类指令
- GPT-4 / Claude：多模态、更长上下文

**自回归生成**

```text
输入：["你", "好"]
预测：["好", "世"]
预测：["世", "界"]
...
```

每次预测下一个 token，概率最高的即为输出（或采样）。

**Tokenization**

- BPE（Byte-Pair Encoding）：子词分割
- 词表大小通常 32k ～ 128k
- 中文 token 效率低于英文（约 1.5-2x 倍）

**上下文窗口与 KV Cache**

- 上下文窗口：模型能"看到"的最大 token 数
- KV Cache：缓存历史 token 的 Key/Value，推理加速
- Flash Attention：节省显存的注意力计算实现

**涌现能力（Emergence）**

- 小模型没有，大模型才有的能力
- 示例：In-context learning、思维链推理、指令遵循

**RLHF（基于人类反馈的强化学习）**

1. 监督微调（SFT）：在人工标注数据上微调
2. 奖励模型（RM）：学习人类偏好打分
3. PPO 强化学习：用奖励信号优化生成策略

### 建议练习

1. 用 `tiktoken` 对中英文句子做 tokenize，观察 token 数差异。
2. 调用 Claude API，通过 `max_tokens` 和 `temperature` 参数感受生成控制。
3. 阅读 Anthropic 或 OpenAI 的模型卡（Model Card），了解训练方式。

### 阶段成果

- 能解释 GPT 为什么能回答问题（预训练 + 指令微调）。
- 理解 token、上下文窗口、温度采样等核心概念。
- 能区分 BERT 和 GPT 的适用场景。

## 阶段 7：LangChain 与 LlamaIndex

- `PromptTemplate`：带变量的提示模板
- `OutputParser`：结构化解析输出
- `Chain`：串联多个组件
- `LCEL`（LangChain Expression Language）：`|` 管道符组合

```python
from langchain_anthropic import ChatAnthropic
from langchain_core.prompts import ChatPromptTemplate

llm = ChatAnthropic(model="claude-sonnet-4-6")
prompt = ChatPromptTemplate.from_template("请用一句话总结：{text}")
parser = StrOutputParser()

chain = prompt | llm | parser
result = chain.invoke({"text": "人工智能正在改变各行各业..."})
```

**Memory（记忆模块）**

- `ConversationBufferMemory`：完整历史
- `ConversationSummaryMemory`：摘要压缩历史
- `ConversationTokenBufferMemory`：按 token 截断

**LlamaIndex 核心概念**

- `Document`：原始文档
- `Node`：分块后的文本单元
- `VectorStoreIndex`：向量索引
- `QueryEngine`：查询引擎

## 阶段 8：RAG 检索增强生成

目标：能独立搭建企业级 RAG 系统，解决 LLM 幻觉和知识过时问题。

### 学习内容

**RAG 解决的问题**

- LLM 训练数据有截止日期
- LLM 对私有文档一无所知
- LLM 会产生幻觉（编造事实）
- RAG = 给 LLM 实时检索相关上下文

**基础 RAG 流程**

```text
离线阶段：
文档 → 分块（Chunking）→ Embedding → 存入向量数据库

在线阶段：
用户提问 → Embedding → 向量检索 → 取 Top-K 文档 → 拼入 Prompt → LLM 生成
```

**文档分块策略**

- 固定大小分块：按字符数切割（简单但会截断语义）
- 递归分块：按段落 → 句子 → 字符递归切割
- 语义分块：相似段落聚合（质量最高）
- 建议：chunk_size=512，overlap=50

**Embedding 模型选型**

| 模型 | 特点 |
|------|------|
| `text-embedding-3-large`（OpenAI）| 效果好，收费 |
| `BGE-M3`（BAAI）| 开源，多语言，中文强 |
| `text2vec-large-chinese` | 中文专用，轻量 |

**向量数据库**

```python
import chromadb

client = chromadb.Client()
collection = client.create_collection("my_docs")

collection.add(
    documents=["文档内容1", "文档内容2"],
    embeddings=[[0.1, 0.2, ...], [0.3, 0.4, ...]],
    ids=["doc1", "doc2"]
)

results = collection.query(query_embeddings=[[0.15, 0.25, ...]], n_results=3)
```

**混合检索**

- 向量检索：语义相似，捕捉同义词
- BM25 关键词检索：精确匹配，捕捉专有名词
- 两者结合（RRF 倒数排名融合）效果最佳

**重排序（Rerank）**

- 检索 Top-20，用 Reranker 精排取 Top-3
- 模型：`BGE-Reranker-v2`、`Cohere Rerank`

**RAG 评估**

- 检索召回率：相关文档是否被检索到
- 答案忠实性（Faithfulness）：答案是否来自检索内容
- 答案相关性（Relevance）：答案是否回答了问题
- 工具：RAGAS 框架

**高级 RAG 技术**

- HyDE（假设性文档嵌入）：先让 LLM 生成假设答案再检索
- Self-RAG：模型自主判断是否需要检索
- Corrective RAG：检索质量差时自动降级到网络搜索
- 父文档检索器：检索小块，返回父块保证上下文完整

### 建议练习

1. 对一批 PDF 文档建立 RAG 系统，实现中文问答。
2. 对比不同 chunk_size（256/512/1024）对检索质量的影响。
3. 加入 Reranker，对比有无 Rerank 的答案质量。
4. 用 RAGAS 评估系统的 Faithfulness 和 Relevance 指标。

## 阶段 9：AI Agent 开发

目标：能开发具备工具调用、多步规划能力的 AI Agent。

### 学习内容

**Agent 的本质**

```text
while not done:
    思考（Thought）：分析当前状态，决定下一步
    行动（Action）：调用工具或生成答案
    观察（Observation）：获取工具执行结果
```

**Function Calling / Tool Use**

```python
import anthropic

tools = [
    {
        "name": "get_weather",
        "description": "获取指定城市的实时天气",
        "input_schema": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名称"}
            },
            "required": ["city"]
        }
    }
]

client = anthropic.Anthropic()
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "北京今天天气怎么样？"}]
)

# 如果模型决定调用工具
if response.stop_reason == "tool_use":
    tool_use = next(b for b in response.content if b.type == "tool_use")
    print(f"调用工具：{tool_use.name}，参数：{tool_use.input}")
```

**ReAct Agent 实现**

1. 定义工具集（搜索、计算、数据库查询等）
2. 构造 ReAct 提示模板
3. 循环执行：LLM 决策 → 工具执行 → 结果回传
4. 设置最大步数防止死循环

**LangGraph（推荐）**

- 将 Agent 工作流建模为有向图（DAG/带环图）
- 节点：LLM 调用、工具调用、条件判断
- 边：控制流转，支持循环
- 适合复杂、有状态的 Agent

**多 Agent 协作**

```text
Manager Agent
├── Research Agent（负责信息检索）
├── Analysis Agent（负责数据分析）
└── Writer Agent（负责生成报告）
```

## 阶段 10：模型微调与部署

目标：能对开源模型做指令微调，并部署为可调用的推理服务。

### 学习内容

**为什么微调**

- 通用模型不熟悉特定领域术语和风格
- 需要强制模型遵循特定输出格式
- 降低推理成本（小模型替代大模型）

**参数高效微调（PEFT）**

**LoRA 原理：**

```text
原始权重矩阵 W（冻结，不更新）
新增低秩矩阵 A、B（只训练这两个）
实际更新：W' = W + A × B
r << d，大幅减少参数量
```

**QLoRA**：4bit 量化 + LoRA，可在消费级 GPU 上微调 7B 模型。

**微调数据集构建**

```json
[
  {
    "instruction": "将以下文本翻译为正式公文风格",
    "input": "我们公司最近搞了个新项目",
    "output": "本公司近期启动了一项新项目"
  }
]
```

- 数量：通常 500～5000 条高质量数据即可
- 质量 > 数量：宁可 500 条精标，不要 10000 条噪声数据

**使用 HuggingFace 微调**

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import get_peft_model, LoraConfig
from trl import SFTTrainer

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-7B")
lora_config = LoraConfig(r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"])
model = get_peft_model(model, lora_config)

trainer = SFTTrainer(model=model, train_dataset=dataset, ...)
trainer.train()
```

**模型量化**

- INT8：精度损失小，速度快 2x
- INT4：精度有损，速度快 4x，可在 8GB 显存运行 7B 模型
- GGUF：CPU 推理格式，配合 llama.cpp 使用

**推理服务部署**

- **Ollama**：本地一键部署，适合开发测试
  ```bash
  ollama run qwen2.5:7b
  ```

- **vLLM**：生产级推理引擎，支持 PagedAttention 和连续批处理
  ```bash
  python -m vllm.entrypoints.openai.api_server --model Qwen/Qwen2.5-7B
  ```

- **封装为 API**：FastAPI + vLLM，对外提供 OpenAI 兼容接口

### 建议练习

1. 用 QLoRA 微调 Qwen2.5-7B，使其输出特定格式的 JSON。
2. 用 Ollama 在本地跑一个开源模型，对比不同量化版本的速度和质量。
3. 用 vLLM 部署微调后的模型，并用 FastAPI 封装接口。

### 阶段成果

- 能微调开源模型适应特定业务需求。
- 理解量化对模型大小和质量的影响。
- 能部署可调用的本地推理服务。

## 阶段 11：AI 应用架构与工程化

目标：能设计生产级 AI 应用，解决性能、成本、可靠性问题。

### 学习内容

**流式输出（Streaming）**

```python
import anthropic

client = anthropic.Anthropic()
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "写一首诗"}]
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
```

**异步并发**

```python
import asyncio
import anthropic

client = anthropic.AsyncAnthropic()

async def process_item(item):
    response = await client.messages.create(...)
    return response

# 并发处理 10 条请求
results = await asyncio.gather(*[process_item(i) for i in items])
```

**对话历史管理**

- 存储策略：数据库持久化（按 session_id）
- 截断策略：按 token 数截断，保留最近 N 轮
- 压缩策略：用 LLM 定期摘要旧历史

**成本控制**

- Token 计算：提前用 `tiktoken` 估算，避免超额
- Prompt 缓存：Anthropic 支持 `cache_control`，缓存 System Prompt，节省 90% 输入成本
- 模型降级：简单任务用 Haiku，复杂任务用 Sonnet/Opus
- 请求去重：缓存相同问题的答案

**Prompt 缓存示例（Anthropic）**

```python
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "你是专业客服...",
            "cache_control": {"type": "ephemeral"}  # 缓存 System Prompt
        }
    ],
    messages=[{"role": "user", "content": user_input}]
)
```

**可观测性**

- 日志：记录每次 LLM 调用的输入、输出、延迟、Token 数
- 追踪：LangSmith（LangChain 生态）、Arize Phoenix（开源）
- 监控指标：P50/P95 延迟、错误率、Token 使用量、成本

**安全与合规**

- 内容过滤：检测有害输出，拒绝服务
- PII 脱敏：用户个人信息发送给 LLM 前脱敏
- 审计日志：记录所有 LLM 交互，满足合规要求

**典型 AI 应用架构**

```text
用户请求
    ↓
API 网关（鉴权、限流）
    ↓
AI 服务（FastAPI + asyncio）
    ├── 对话历史管理（Redis）
    ├── RAG 检索（向量数据库）
    ├── LLM 调用（Anthropic / OpenAI）
    └── 结果缓存（Redis）
    ↓
可观测性平台（LangSmith）
```

### 建议练习

1. 为 RAG 问答系统接入 LangSmith，追踪每次检索和生成的过程。
2. 实现 Prompt 缓存，测量 Token 节省比例。
3. 实现流式输出接口，前端实时显示生成内容。
4. 压测 LLM 接口，找到系统瓶颈并优化。

### 阶段成果

- 能设计可观测、可维护的生产级 AI 应用。
- 能有效控制 LLM 调用成本。
- 理解 AI 应用与普通 Web 应用在架构上的差异。
