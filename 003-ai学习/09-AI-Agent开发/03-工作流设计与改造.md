# 工作流（Workflow）的设计与改造

> **一句话先说清楚**：Workflow 是"你提前把步骤画死、让 LLM 在每一格里干活"的固定流水线；Agent 是"你只给目标、让 LLM 自己决定走哪步"的自由探索者。**大多数真实业务该用 Workflow，而不是 Agent**——因为它可控、可测、可改。这篇讲两件事：怎么从零设计一个 Workflow，以及上线后怎么一轮轮把它改好。

---

## 第一部分：Workflow 到底是什么，凭什么优先用它（总）

### Workflow vs Agent，一张图讲清

```
Workflow（工作流）：路径由你定，LLM 只在节点里干活
   ┌─────┐   ┌─────┐   ┌─────┐
   │ 步骤1│ → │ 步骤2│ → │ 步骤3│      流程图是写死的
   └─────┘   └─────┘   └─────┘

Agent（智能体）：路径由 LLM 自己决定
   ┌─────┐
   │ LLM │ ⇄ 工具       下一步走哪、循环几次，模型说了算
   └─────┘
```

| 维度 | Workflow | Agent |
|------|----------|-------|
| 路径 | **人提前定好**，固定 | LLM 运行时自己决定 |
| 可控性 | 高，每步都可预测 | 低，可能跑飞 |
| 可测试 | 容易，每节点单独测 | 难，路径不固定 |
| 成本 | 可预估 | 不可预估（可能循环很多次） |
| 适合场景 | 步骤明确的业务 | 开放、探索性任务 |

### 一条最重要的工程原则

> **能用 Workflow 解决的，就别用 Agent。**

很多人一上来就想做"全自主 Agent"，结果上线后发现：行为不可预测、出了问题没法定位、成本失控。

真实世界里 90% 的 LLM 应用——客服分流、文档处理、内容生成、数据提取——**步骤其实是明确的**。这些就该用 Workflow：你把流程画死，每一格让 LLM 发挥，整体可控、可测、可改。

只有当"下一步该干什么"本身需要模型动态判断、且无法提前枚举时，才升级到 Agent。

---

### 五种经典 Workflow 模式（必须掌握）

这是 Anthropic《Building Effective Agents》提炼的五种基础模式，几乎所有复杂 Workflow 都是它们的组合。

**模式1：Prompt Chaining（提示链）—— 把大任务拆成串行小步**

```
输入 → [LLM:生成大纲] → [LLM:扩写正文] → [LLM:润色] → 输出
```
适合：任务能清晰拆成"一步接一步"的顺序步骤。
好处：每步只干一件事，质量比"一个 prompt 全干完"高得多。

**模式2：Routing（路由）—— 先分类，再分流到专门处理**

```
                 ┌→ [退款处理]
输入 → [分类器] ─┼→ [技术支持]
                 └→ [账单咨询]
```
适合：输入有明显类别，不同类别处理方式不同（如客服系统）。
好处：每条分支用专门优化的 prompt，比"一个万能 prompt"准。

**模式3：Parallelization（并行）—— 同时干，再汇总**

```
        ┌→ [LLM:从安全角度审] ─┐
输入 ──┼→ [LLM:从性能角度审] ─┼→ [汇总] → 输出
        └→ [LLM:从风格角度审] ─┘
```
适合：一个任务能拆成互不依赖的子任务（如多角度评审、多项检查）。
好处：更快（并发），且每个角度独立，互不干扰。

**模式4：Orchestrator-Workers（编排者-工人）—— 动态拆任务**

```
输入 → [编排者:看情况拆成N个子任务] → [多个工人并行做] → [汇总]
```
适合：子任务数量/类型事先不确定（如"改这个代码库的所有相关文件"——改几个文件运行时才知道）。
和并行的区别：并行的子任务是**写死的**，这里是**编排者运行时动态决定的**。

**模式5：Evaluator-Optimizer（评估-优化循环）—— 自我迭代**

```
输入 → [生成者:产出] → [评估者:打分+提意见] →满意? ─是→ 输出
                ↑__________________________└─否，带着意见重做
```
适合：有明确评判标准、且第一次往往不够好的任务（如翻译、写作）。
好处：用"出题+批改"的闭环，质量逐轮提升。

> **记住**：这五种是积木。真实系统常常是"先路由，分支里用提示链，某一步再套评估优化循环"的组合。

## 第二部分：从 0 设计一个 Workflow 的完整过程（分）

下面用一个真实例子走完整个设计流程：**做一个"用户反馈自动处理"系统**——收到用户反馈后，自动分类、提取要点、生成回复草稿、并对负面反馈升级告警。

### 第1步：把业务流程画出来（先别碰代码）

设计 Workflow，**永远从画流程图开始，不是从写 prompt 开始**。

```
用户反馈
   ↓
[1. 分类]  → 是 bug / 功能建议 / 表扬 / 投诉？
   ↓
[2. 提取结构化信息]  → 产品模块、严重程度、关键诉求
   ↓
[3. 路由]
   ├─ 投诉/严重bug → [4a. 生成升级告警 + 安抚回复]
   └─ 其他          → [4b. 生成常规回复草稿]
   ↓
[5. 汇总输出]  → 结构化结果 + 回复草稿
```

画图时问自己三个问题：
1. **哪些步骤必须串行**（后一步依赖前一步的结果）？→ 用提示链
2. **哪里需要分情况处理**？→ 用路由
3. **哪些步骤互不依赖、可以同时做**？→ 用并行

### 第2步：定义贯穿全流程的"状态"

Workflow 的每一步都要读写一个共享的状态对象。先把它定义清楚：

```python
from dataclasses import dataclass, field
from typing import Literal

@dataclass
class FeedbackState:
    # 输入
    raw_text: str

    # 各步骤逐渐填充的字段
    category: Literal["bug", "feature", "praise", "complaint"] | None = None
    module: str = ""                 # 涉及的产品模块
    severity: Literal["low", "medium", "high"] | None = None
    key_points: list[str] = field(default_factory=list)
    reply_draft: str = ""
    need_escalation: bool = False
```

> 状态就是流水线上传递的"工件托盘"，每个工位往上面放加工好的零件。

### 第3步：每个节点写成一个独立函数

**关键原则：一个节点 = 一个单一职责的函数**。不要让一个节点干两件事。

```python
from anthropic import Anthropic
import json

client = Anthropic()

def classify(state: FeedbackState) -> FeedbackState:
    """节点1：分类"""
    prompt = f"""把下面的用户反馈分类为 bug/feature/praise/complaint 之一，只输出类别词。

反馈：{state.raw_text}"""
    resp = client.messages.create(
        model="claude-haiku-4-5",   # 简单分类用便宜快的模型
        max_tokens=10, temperature=0,
        messages=[{"role": "user", "content": prompt}]
    )
    state.category = resp.content[0].text.strip()
    return state


def extract_info(state: FeedbackState) -> FeedbackState:
    """节点2：提取结构化信息"""
    prompt = f"""从反馈中提取信息，输出 JSON：
{{"module": "涉及的产品模块", "severity": "low/medium/high", "key_points": ["要点1","要点2"]}}

反馈：{state.raw_text}"""
    resp = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=300, temperature=0,
        messages=[{"role": "user", "content": prompt}]
    )
    data = json.loads(resp.content[0].text)
    state.module = data["module"]
    state.severity = data["severity"]
    state.key_points = data["key_points"]
    return state


def route(state: FeedbackState) -> str:
    """节点3：路由——返回下一步走哪条分支"""
    if state.category == "complaint" or state.severity == "high":
        state.need_escalation = True
        return "urgent"
    return "normal"


def reply_urgent(state: FeedbackState) -> FeedbackState:
    """节点4a：紧急情况——安抚 + 标记升级"""
    prompt = f"""用户有严重不满，写一段诚恳的安抚回复（不超过100字）。
反馈：{state.raw_text}
要点：{state.key_points}"""
    resp = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=300, temperature=0.7,
        messages=[{"role": "user", "content": prompt}]
    )
    state.reply_draft = resp.content[0].text
    return state


def reply_normal(state: FeedbackState) -> FeedbackState:
    """节点4b：常规回复"""
    prompt = f"""写一段友好的回复草稿（不超过100字）。
反馈：{state.raw_text}"""
    resp = client.messages.create(
        model="claude-haiku-4-5", max_tokens=300, temperature=0.7,
        messages=[{"role": "user", "content": prompt}]
    )
    state.reply_draft = resp.content[0].text
    return state
```

### 第4步：把节点串成流水线

```python
def run_workflow(raw_text: str) -> FeedbackState:
    state = FeedbackState(raw_text=raw_text)

    state = classify(state)             # 步骤1
    state = extract_info(state)         # 步骤2

    branch = route(state)               # 步骤3：决定分支
    if branch == "urgent":
        state = reply_urgent(state)     # 步骤4a
    else:
        state = reply_normal(state)     # 步骤4b

    return state                        # 步骤5：返回完整结果


# 用一下
result = run_workflow("你们最新版本一打开就闪退，我数据全没了，太离谱了！")
print(f"分类：{result.category}")
print(f"严重程度：{result.severity}")
print(f"需要升级：{result.need_escalation}")
print(f"回复草稿：{result.reply_draft}")
```

这就是一个完整可跑的 Workflow。**注意它有多透明**：每一步的输入输出都看得见，出问题能精确定位到哪个节点。这正是 Workflow 相对 Agent 的核心优势。

## 第三部分：用 LangGraph 重写（分）

手写版本节点一多，路由逻辑会越来越乱。LangGraph 专门用来管理这种"节点 + 边"的图结构。

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, Literal

class State(TypedDict):
    raw_text: str
    category: str
    severity: str
    reply_draft: str
    need_escalation: bool

# 节点函数（返回要更新的字段）
def classify_node(state: State) -> dict:
    # ...调用 LLM...
    return {"category": "complaint"}

def extract_node(state: State) -> dict:
    return {"severity": "high"}

def reply_urgent_node(state: State) -> dict:
    return {"reply_draft": "...", "need_escalation": True}

def reply_normal_node(state: State) -> dict:
    return {"reply_draft": "..."}

# 条件路由函数
def route_condition(state: State) -> Literal["urgent", "normal"]:
    if state["category"] == "complaint" or state["severity"] == "high":
        return "urgent"
    return "normal"

# 搭图
graph = StateGraph(State)
graph.add_node("classify", classify_node)
graph.add_node("extract", extract_node)
graph.add_node("reply_urgent", reply_urgent_node)
graph.add_node("reply_normal", reply_normal_node)

graph.set_entry_point("classify")
graph.add_edge("classify", "extract")
graph.add_conditional_edges("extract", route_condition, {
    "urgent": "reply_urgent",
    "normal": "reply_normal",
})
graph.add_edge("reply_urgent", END)
graph.add_edge("reply_normal", END)

app = graph.compile()
result = app.invoke({"raw_text": "打开就闪退！"})
```

LangGraph 的价值：
- **流程可视化**：能把图画出来，一眼看清结构
- **状态自动管理**：不用手动传 state
- **天然支持**断点续跑、人工介入、并行分支等高级能力

> 但别为了用框架而用框架。**节点少（3~4个）时，手写更清晰**；节点多、有复杂分支和循环时，再上 LangGraph。

## 第四部分：改造 —— 上线后怎么一轮轮把 Workflow 改好（分）⭐

设计出第一版只是开始。真正的功夫在**上线后的迭代改造**。这一部分是很多教程不讲、但最值钱的。

### 改造的总原则：先测量，再改造

> **没有评估数据，就不要改 Workflow。** 否则你改完根本不知道是变好了还是变坏了。

改造前必须先有两样东西：
1. **一批真实测试用例**（输入 + 期望输出，20~50 条起步）
2. **一套评估指标**（准确率、通过率、人工评分等）

```python
# 最简单的评估骨架
test_cases = [
    {"input": "打开就闪退", "expect_category": "bug", "expect_severity": "high"},
    {"input": "希望加个夜间模式", "expect_category": "feature"},
    # ... 更多真实案例
]

def evaluate(workflow_fn):
    correct = 0
    for case in test_cases:
        result = workflow_fn(case["input"])
        if result.category == case["expect_category"]:
            correct += 1
    return correct / len(test_cases)

baseline_score = evaluate(run_workflow)
print(f"当前准确率：{baseline_score:.1%}")     # 这是你的基线
```

有了基线，每次改造后重新跑一遍，**用数字证明改好了**。

---

### 改造场景1：某个节点质量不行 → 局部优化，别动全局

Workflow 最大的好处就是**问题可以隔离**。先定位是哪个节点拖后腿：

```python
# 单独测分类节点的准确率
def eval_classify():
    correct = 0
    for case in test_cases:
        s = FeedbackState(raw_text=case["input"])
        s = classify(s)
        if s.category == case["expect_category"]:
            correct += 1
    return correct / len(test_cases)

print(f"分类节点准确率：{eval_classify():.1%}")   # 比如只有 70%，那问题就在这
```

定位到分类节点后，**只改这一个节点的 prompt**，常见手段：
- 加 Few-shot 示例（给几个分类样例）
- 把模糊的类别定义写清楚
- 换更强的模型（haiku → sonnet）

改完只重测这个节点，达标了再看整体。**不要一次动多个节点**，否则分不清是谁的功劳/锅。

---

### 改造场景2：流程结构有问题 → 重组节点

有时问题不在单个节点，而在**流程编排**本身。常见的几种结构性改造：

**① 一个节点干太多 → 拆开**

```
改造前：[一个节点同时做"分类+提取+判断严重程度"]   ← 又难调又不准
改造后：[分类] → [提取] → [判断严重程度]            ← 各管一段，每段都好优化
```

**② 串行太慢 → 改并行**

```
改造前：[安全审查] → [性能审查] → [风格审查]    （串行，3倍耗时）
改造后：三个审查并行 → 汇总                       （并发，1倍耗时）
```

**③ 质量不稳 → 加评估优化循环**

```
改造前：[生成回复] → 直接输出                    （时好时坏）
改造后：[生成回复] → [评估打分] →不合格→ 重写     （兜住下限）
```

**④ 简单输入也走全流程 → 加快速通道**

```
改造前：所有反馈都走完整5步
改造后：[快速判断] → 简单的表扬类 → 直接模板回复（省钱省时）
                  → 复杂的 → 走完整流程
```

---

### 改造场景3：从 Workflow 升级到 Agent（谨慎）

当你发现 **"下一步该走哪，越来越难用固定规则描述"** 时，才考虑给某个环节引入 Agent 的自主性。

```
信号：你的 route() 函数里 if-else 分支爆炸到几十个，还总有漏的情况
     → 说明路径太复杂，固定规则覆盖不全
     → 可以把"决定下一步"这件事交给 LLM 动态判断（局部 Agent 化）
```

但记住：**只在必要的那一个环节引入自主性，其余部分保持 Workflow 的可控**。不要整个推倒重做成全自主 Agent。这种"大部分固定 + 关键处自主"的混合架构，才是生产级系统的常态。

---

### 改造的迭代节奏

```
1. 跑评估，拿到基线分数
2. 看哪里最差（定位到具体节点 / 结构问题）
3. 只改一个地方
4. 重新跑评估
5. 变好了 → 保留；变差了 → 回滚
6. 回到第2步，改下一个最差的点
```

**一次只改一个变量**，是 Workflow 改造的铁律。这和调试代码、做科学实验是同一个道理。

## 第五部分：设计与改造的核心原则（总）

### 设计阶段的检查清单

- [ ] 先画流程图，再写代码
- [ ] 每个节点单一职责，不贪多
- [ ] 状态对象定义清晰，字段含义明确
- [ ] 简单节点用便宜模型（haiku），关键节点用强模型（sonnet/opus）
- [ ] 优先 Workflow，确实需要才上 Agent

### 改造阶段的铁律

- [ ] 没有评估数据，不动 Workflow
- [ ] 先定位到具体节点/结构，再改
- [ ] 一次只改一个地方，用数字验证
- [ ] 变差就回滚，别舍不得

### 五种模式速查

| 模式 | 一句话 | 何时用 |
|------|--------|--------|
| 提示链 | 拆成串行小步 | 步骤清晰可顺序拆分 |
| 路由 | 先分类再分流 | 输入有明显类别 |
| 并行 | 同时干再汇总 | 子任务互不依赖 |
| 编排者-工人 | 动态拆任务 | 子任务数运行时才知道 |
| 评估优化 | 出题+批改循环 | 有评判标准、需迭代提质 |

### 一句话总结

> **设计 Workflow，是把一个复杂任务"拆"成一串可控、可测的小格子；改造 Workflow，是拿着评估数据，一格一格地把它打磨好。**
>
> Agent 给你自由，Workflow 给你掌控——而生产环境，掌控往往比自由更值钱。

---

## 本节重点

- Workflow = 路径写死、LLM 在节点里干活；Agent = 路径由 LLM 自定
- **能用 Workflow 就别用 Agent**——可控、可测、可改、成本可预估
- 五种基础模式：提示链 / 路由 / 并行 / 编排者-工人 / 评估优化，复杂系统是它们的组合
- 设计流程：**先画图 → 定状态 → 每节点单一职责 → 串起来**
- 改造铁律：**没评估数据不动手；先定位再改；一次只改一处；变差就回滚**
- 节点少手写更清晰，节点多再上 LangGraph
- 生产级系统常是"大部分 Workflow + 关键处局部 Agent"的混合架构
