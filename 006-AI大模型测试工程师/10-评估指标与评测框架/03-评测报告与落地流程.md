# 03 评测报告与落地流程

## 学习目标

这一节学习如何把评测落地成流程和报告。

你需要掌握：

- 评测数据怎么管理。
- 评测流程怎么设计。
- 报告应该包含什么。
- 如何接入 CI 回归。

## 标准评测流程

```text
确定测试目标
    ↓
准备测试集
    ↓
运行被测 AI 应用
    ↓
收集输出
    ↓
执行规则评估
    ↓
执行 LLM Judge 或框架评估
    ↓
人工抽样复核
    ↓
生成报告
    ↓
失败样本进入回归集
```

## 测试集管理

推荐按业务分类：

```text
datasets/
├── prompt_cases.json
├── rag_cases.json
├── agent_cases.json
├── safety_cases.json
└── regression_cases.json
```

每条样本建议包含：

```json
{
  "id": "case_001",
  "input": "用户问题",
  "expected_keywords": ["关键词1", "关键词2"],
  "expected_source": "文档.md",
  "category": "RAG",
  "risk_level": "high"
}
```

## 报告必须包含什么

### 1. 测试范围

- 测试对象
- 模型版本
- Prompt 版本
- 知识库版本
- 测试集版本
- 测试时间

### 2. 总体结果

| 指标 | 数值 |
|------|------|
| 总用例数 | 100 |
| 通过数 | 86 |
| 失败数 | 14 |
| 通过率 | 86% |
| 平均响应时间 | 2.1s |
| 平均 Token 成本 | 1200 |

### 3. 分模块结果

| 模块 | 通过率 |
|------|--------|
| Prompt | 92% |
| RAG | 84% |
| Agent | 78% |
| Safety | 88% |

### 4. 失败分类

| 失败类型 | 数量 |
|----------|------|
| 格式错误 | 3 |
| 检索未命中 | 5 |
| 幻觉 | 4 |
| 工具参数错误 | 2 |

### 5. 典型失败样本

每个失败样本记录：

- 用例 ID
- 输入
- 期望结果
- 实际输出
- 失败原因
- 修复建议

## CI 落地方式

建议分两层：

### 快速门禁

每次提交都跑：

- 接口可用性
- JSON 格式
- 高风险回归用例
- 安全 smoke 测试

### 完整评测

每天或发布前跑：

- 全量 Prompt 测试
- 全量 RAG 测试
- 全量 Agent 测试
- LLM Judge 评估
- 成本和延迟统计

## 通过标准示例

```text
接口成功率 >= 99%
JSON 格式通过率 = 100%
高风险用例通过率 = 100%
RAG Faithfulness >= 0.85
Agent 工具调用正确率 >= 0.9
安全红队严重问题 = 0
平均响应时间 <= 3s
```

这些阈值不是固定标准，需要按业务风险调整。

## 本节重点

- AI 评测要有测试集版本、Prompt 版本、模型版本。
- 报告要能定位失败原因。
- CI 中要区分快速门禁和完整评测。
- 失败样本要沉淀到回归集。
- 评测不是一次性活动，而是持续质量体系。
