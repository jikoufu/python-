## 一、基础概念

### ⭐ Q1：负载测试和压力测试有什么区别？

**答**：负载测试是在**预期负载**下验证系统性能是否达标，比如"日常高峰 500 并发下响应时间是否 ≤200ms"。压力测试是**不断加压直到系统崩溃**，目的是找系统的上限和崩溃点。一个看"正常情况达不达标"，一个看"极限在哪、什么时候崩"。

### ⭐ Q2：常用的性能指标有哪些？

**答**：核心是 TPS（每秒事务数，衡量吞吐）、响应时间 RT（尤其看 P99 而不是平均值）、并发数、吞吐量、错误率。还有资源指标 CPU、内存、IO、网络使用率。

### Q3：为什么响应时间要看 P99 不看平均值？

**答**：平均值会被大量快请求拉低，掩盖少数慢得离谱的请求。P99 表示 99% 的请求都快于这个值，更能反映真实的用户体验和尾部延迟。

### ⭐ Q4：TPS、并发数、响应时间什么关系？

**答**：并发数 ≈ TPS × 响应时间（利特尔法则）。所以同样并发下，响应时间越短，TPS 越高。加压时 TPS 先随并发线性上升，到拐点达到峰值（系统上限），再加压 TPS 反降、RT 暴涨——那个拐点就是系统的最大处理能力。

### Q5：怎么确定性能测试的目标/通过标准？

**答**：根据业务量估算峰值 TPS（常用二八原则），然后定量化门槛，比如"500 并发下 TPS≥800、P99≤200ms、错误率≤0.1%、CPU≤80%"。目标必须可测、可验收，不能是"要很快"这种模糊说法。

## 二、JMeter 工具

### ⭐ Q6：JMeter 的核心组件有哪些？

**答**：测试计划下面是线程组（控制并发和加压方式），线程组里有取样器（发请求，HTTP Request 最常用）、断言（校验响应）、定时器（控制请求间隔）、配置元件（CSV 参数、HTTP 默认值）、前后置处理器（参数化、关联）、监听器（看结果）。

### ⭐ Q7：线程组的三个参数是什么？Ramp-Up 设 0 有什么问题？

**答**：线程数（虚拟用户数）、Ramp-Up（多久把用户全部启动）、循环次数。Ramp-Up 设 0 表示瞬间启动所有线程，既不真实，也容易把发压机自己搞崩，应该给一个爬坡时间模拟用户逐渐涌入。

### ⭐ Q8：参数化和关联分别是什么？

**答**：参数化是让不同用户使用不同测试数据（如不同账号），用 CSV Data Set Config 实现。关联是把前一个请求响应里的动态值提取出来给后续请求用（如登录拿 token 后续请求带上），用 JSON 提取器或正则提取器实现。

### Q9：测秒杀的瞬时并发怎么模拟？

**答**：用同步定时器（Synchronizing Timer，集合点），让所有线程攒齐后同一瞬间一起发请求，模拟秒杀的瞬时高并发。

### ⭐ Q10：为什么正式压测要用命令行而不是 GUI？

**答**：GUI 模式很消耗资源，会影响发压能力和结果准确性。GUI 只用来调试脚本，正式压测用命令行 `jmeter -n -t 脚本.jmx -l 结果.jtl -e -o 报告目录`，非 GUI 模式发压能力强，还能自动生成 HTML 报告。

### Q11：单机压不上去怎么办？

**答**：先排查是不是发压机自己成了瓶颈（客户端 CPU、内存、网卡打满）。确认不是后，用分布式压测：一台 Master 做调度和汇总，多台 Slave 一起发压。

## 三、瓶颈分析（重点中的重点）

### ⭐ Q12：你怎么定位性能瓶颈？

**答**：自上而下分层排查。先确认不是发压机的问题，再看现象定方向（TPS 上不去 / RT 高 / 错误率高方向不同），然后逐层看监控——客户端、网络、网关、应用（CPU/内存/GC/线程池/连接池）、中间件、数据库、磁盘 IO——找到资源被打满或异常的那一层，深入找根因，优化后再压一遍验证。核心是用数据定位，不靠猜。

### ⭐ Q13：TPS 上不去，但 CPU、内存都不高，可能是什么原因？

**答**：这说明瓶颈不是算力，而是"等待"。常见原因：数据库连接池/线程池太小导致请求排队、有锁竞争（大量请求抢同一行数据串行执行）、卡在某个慢的外部依赖、同步串行调用。要看线程栈，大量线程 BLOCKED 在等连接或等锁就能确认。

### Q14：频繁 Full GC 会有什么表现？怎么排查？

**答**：Full GC 会 Stop The World 暂停应用，表现为 TPS 周期性掉零、RT 出现尖刺、吞吐抖动。用 `jstat -gcutil` 看 Full GC 频率和耗时，如果频繁增长就有问题。原因可能是堆太小、对象创建过多、大对象或内存泄漏。

### ⭐ Q15：数据库瓶颈一般查什么？

**答**：三方面。慢查询（开慢查询日志，EXPLAIN 看有没有走索引、是否全表扫描，常因缺索引、SELECT *、深分页）；连接池（太小请求排队，泄漏会耗尽）；锁（行锁/表锁竞争导致串行，死锁导致超时）。

### Q16：CPU 的 wa（IO wait）很高说明什么？

**答**：说明 CPU 在等 IO（磁盘或网络），自己空转干等，真正的瓶颈在 IO 而不是 CPU。要去查磁盘 IO（iostat 看 %util、await）或网络。

### ⭐ Q17（案例题）：讲一个你定位性能问题的经历。

**答**（黄金模板）：

> 现象：订单接口压测，并发到 200 时 TPS 卡在 300 不涨，RT 从 100ms 涨到 2s，但应用 CPU 才 40%。
> 排查：先确认发压机正常；CPU 不高但 TPS 上不去，怀疑等待；看线程栈发现大量线程阻塞在等数据库连接；查连接池发现最大 20 已占满；查慢查询发现订单 SQL 没走索引，单条 80ms 占着连接不放。
> 根因：慢 SQL 拉长了占用连接的时间 + 连接池太小，导致请求排队。
> 优化：加索引（SQL 80ms→5ms）+ 连接池 20→50。
> 结果：TPS 从 300 提升到 1500，RT 回到 80ms。

**要点**：一定要"现象→排查→根因→优化→数据验证"全套，且有前后数据对比。哪怕是模拟项目，也要这样讲。

## 四、微服务

### ⭐ Q18：微服务怎么做性能测试？

**答**：分三个层次。先单服务压测摸清每个服务的容量基线；再做链路压测，压完整业务路径，用分布式追踪定位链路里哪个服务是瓶颈；最后（大促前）做全链路压测，在接近生产环境验证整体容量。核心思想是木桶效应——整体性能由最慢的服务决定，优化要先找最短板。

### Q19：微服务里整体慢，怎么知道是哪个服务慢？

**答**：用分布式追踪。给每个请求打全局 TraceID 贯穿所有服务，每个服务记录自己的耗时 Span，最后能画出完整调用链，一眼看出哪一环耗时最多。常用工具 SkyWalking、Jaeger、Zipkin。

### Q20：什么是全链路压测？怎么不污染生产数据？

**答**：全链路压测是在接近生产或生产环境，用真实流量规模压整个系统，大厂大促前常用。靠两点避免污染：流量染色（给压测请求打标记并随调用链透传，让系统识别压测流量）和影子库/影子表（压测数据写到专门的影子表，不碰真实数据）。

### Q21：可观测性三件套是什么？

**答**：Metrics（指标，如 QPS、RT、资源使用率）、Logging（日志）、Tracing（分布式追踪）。三者配合，指标发现异常、追踪定位是哪个环节、日志看具体细节。

## 五、大模型性能（AI 岗位特色）

### ⭐ Q22：大模型接口性能测试和普通接口有什么不同？

**答**：三点。一是慢，响应是秒级流式输出而非毫秒级一次性返回；二是指标不同，除了 TPS/RT 更要看 TTFT（首 token 延迟）和吐字速度，因为流式体验主要看这两个；三是贵，按 token 计费，压测本身就在烧钱，要先估算成本，并发也不能按 Web 接口那样设几千（GPU 资源有限）。

### Q23：大模型调用成本怎么测？怎么优化？

**答**：单次成本 = 输入 token × 输入单价 + 输出 token × 输出单价。测试关注单请求平均 token 和成本、高峰总成本预估。优化手段有模型分层（简单任务用小模型）、prompt 缓存、控制 max_tokens、精简 prompt、结果缓存——测试要验证这些优化确实降低了 token 消耗和成本，并给出"质量达标前提下性能成本最优"的带数据结论。

## 高频必考速记（面试前过一遍）

1. 负载 vs 压力：达标验证 vs 找崩溃点
2. 指标：TPS、RT(看P99)、并发、错误率
3. 公式：并发 ≈ TPS × RT
4. 加压三阶段：线性 → 拐点(上限) → 崩溃
5. JMeter 线程组：线程数、Ramp-Up、循环
6. 正式压测：命令行 `-n -t -l -e -o`
7. 瓶颈定位：自上而下分层，先排除发压机
8. 经典坑：TPS低+资源不高 = 锁/连接池/等待
9. DB 三查：慢查询、连接池、锁
10. 微服务：单服务→链路→全链路，木桶效应，分布式追踪
11. 全链路压测：流量染色 + 影子库
12. 大模型：看 TTFT、吐字速度，按 token 算成本
13. 案例题黄金模板：现象→排查→根因→优化→数据验证
