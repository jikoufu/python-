# 02 JMeter 实战

## 学习目标

学完这一节，你需要能：

- 说清 JMeter 是什么、核心组件有哪些
- 搭一个完整的压测脚本（线程组 + 请求 + 断言 + 监听器）
- 用好参数化、关联、思考时间、集合点
- 看懂聚合报告里的每个字段
- 用命令行（非 GUI）跑压测、生成 HTML 报告
- 知道分布式压测怎么做

---

## 一、JMeter 是什么

JMeter 是 Apache 出的**开源性能测试工具**，纯 Java，能模拟大量用户对服务器发请求，测接口、Web、数据库等的性能。

为什么用它：

```text
- 免费开源，行业标配
- 支持 HTTP/HTTPS、WebSocket、JDBC、JMS 等多种协议
- 图形化录脚本，也能命令行跑
- 插件生态丰富
- 能做分布式压测，单机不够多机一起压
```

---

## 二、JMeter 核心组件（必须理解）

JMeter 脚本是一棵"测试计划树"，由这些组件搭起来：

```text
测试计划 (Test Plan)
└── 线程组 (Thread Group)           ← 模拟多少用户、怎么加压
    ├── 配置元件 (Config Element)    ← 全局配置，如 HTTP 默认值、CSV 数据
    ├── 前置处理器 (Pre Processor)   ← 请求前做的事
    ├── 取样器 (Sampler)             ← 真正发的请求，如 HTTP Request
    ├── 后置处理器 (Post Processor)  ← 请求后提取数据（关联）
    ├── 断言 (Assertion)             ← 判断响应对不对
    ├── 定时器 (Timer)               ← 控制请求间隔（思考时间）
    └── 监听器 (Listener)            ← 收集和展示结果
```

### 最关键的几个组件

| 组件 | 作用 | 通俗解释 |
|------|------|---------|
| **线程组** | 设置并发用户数、加压方式 | "派多少人去压、怎么加人" |
| **取样器 Sampler** | 发请求（HTTP Request 最常用） | "具体要请求哪个接口" |
| **断言 Assertion** | 校验响应是否符合预期 | "返回对不对，别只看快不快" |
| **定时器 Timer** | 加入请求间隔 | "用户不会一秒点 100 下，要模拟真实节奏" |
| **监听器 Listener** | 看结果（聚合报告等） | "看压测出来啥样" |
| **配置元件** | CSV 参数、HTTP 默认值等 | "公共配置和测试数据" |

## 三、线程组：压测的发动机

线程组三个核心参数：

```text
- 线程数 (Number of Threads)：    模拟多少个虚拟用户
- Ramp-Up 时间 (秒)：             多久把这些用户全部启动起来
- 循环次数 (Loop Count)：         每个用户重复执行几次
```

举例：

```text
线程数 = 100，Ramp-Up = 10 秒，循环 = 永远
含义：10 秒内逐步启动 100 个用户（每秒约 10 个），然后持续压
```

> **Ramp-Up 不要设成 0**！瞬间启动几百个线程不真实，还可能把客户端自己搞崩。要给一个爬坡时间，模拟用户逐渐涌入。

### 加压策略

- **递增加压**：Ramp-Up 拉长，观察 TPS 随并发变化，找拐点
- **阶梯加压**：用插件（Concurrency Thread Group / Stepping Thread Group）一段一段加并发
- **稳定压测**：固定并发跑很久，测稳定性

## 四、四个必会技能

### 1. 参数化（让每个用户用不同数据）

不能 100 个用户都用同一个账号登录。用 **CSV Data Set Config** 读数据文件：

```text
准备 users.csv：
  user1,pass1
  user2,pass2
  ...

配置 CSV Data Set Config：
  变量名：username,password
  在请求里用 ${username}、${password} 引用
```

### 2. 关联（提取上一个响应的数据给下一个用）

典型场景：登录拿到 token，后续请求要带这个 token。用**后置处理器**提取：

```text
JSON 提取器：从登录响应里提取 $.data.token → 存为变量 ${token}
后续请求在 Header 里用 ${token}
```

常用提取器：JSON Extractor（JSON 响应）、正则表达式提取器（任意文本）。

### 3. 思考时间（模拟真实用户节奏）

真人不会连续猛点，用**定时器**加间隔：

```text
固定定时器：每个请求后固定等 1 秒
高斯随机定时器：在一个范围内随机等待（更真实）
```

### 4. 集合点（让用户"攒齐了一起发"）

测"秒杀"这种瞬时高并发，要让所有用户卡在同一刻一起发请求。用 **Synchronizing Timer（同步定时器）**：

```text
设置"集合 100 个线程后一起释放" → 模拟 100 人同一瞬间抢购
```

## 五、看懂聚合报告（Aggregate Report）

压测完最常看的就是聚合报告，逐个字段要看懂：

| 字段 | 含义 | 看什么 |
|------|------|--------|
| Samples | 总请求数 | 跑了多少 |
| Average | 平均响应时间(ms) | 参考，别只看它 |
| Median | 中位数(50%) | 一半请求快于它 |
| 90% / 95% / 99% Line | 分位响应时间 | **重点看 99%（尾部延迟）** |
| Min / Max | 最快/最慢 | Max 太离谱要查 |
| Error % | 错误率 | **飙升=到瓶颈了** |
| Throughput | 吞吐量(每秒请求数) | 约等于 TPS |
| KB/sec | 每秒数据量 | 带宽参考 |

> 看报告的顺序：先看 **Error%**（有没有报错）→ 再看 **Throughput/TPS**（扛多少）→ 再看 **99% Line**（慢不慢）。

## 六、命令行跑压测（重点，正式压测必用）

**正式压测绝不能用 GUI 模式**——GUI 很吃资源，会影响压测结果。GUI 只用来调脚本，跑用命令行。

```bash
# 用 .jmx 脚本跑，输出结果到 result.jtl，并生成 HTML 报告到 report 目录
jmeter -n -t test_plan.jmx -l result.jtl -e -o report
```

参数说明：

```text
-n           非 GUI 模式（no GUI）
-t           指定 .jmx 脚本
-l           结果文件 .jtl
-e           测试结束后生成报告
-o           HTML 报告输出目录（目录必须为空）
```

生成的 HTML 报告很直观，有 TPS 曲线、响应时间分布、错误统计，**适合写进测试报告交差**。

## 七、分布式压测（单机压不动时）

一台机器发压能力有限（受 CPU、内存、网卡限制）。要模拟超大并发，用多台机器一起压：

```text
       ┌── Slave 1（发压机）
Master ─┼── Slave 2（发压机）
(控制机) └── Slave 3（发压机）
        ↓
     目标服务器

Master 不发压，只负责调度和汇总结果
Slave 真正发请求
```

> 面试可能问："单机压不上去怎么办？" 答：先排查是不是发压机自己成了瓶颈（客户端 CPU/网卡打满），确认后用**分布式压测**，多台 Slave 一起发。

## 八、一个完整脚本的搭建步骤

把上面串起来，搭一个登录+下单的压测脚本：

```text
1. 新建测试计划
2. 加 HTTP 请求默认值（配置 IP、端口、协议）—— 配置元件
3. 加线程组（100 线程，Ramp-Up 30 秒，循环 10 次）
4. 加 CSV Data Set Config（读 users.csv 参数化账号）
5. 加 HTTP Request：登录接口
6. 加 JSON 提取器：从登录响应提取 token —— 关联
7. 加 HTTP 信息头管理器：把 ${token} 放进后续请求 Header
8. 加 HTTP Request：下单接口（用 ${token}）
9. 加响应断言：校验返回码/字段 —— 断言
10. 加高斯定时器：模拟思考时间
11. 调试通过后，命令行跑：jmeter -n -t xxx.jmx -l r.jtl -e -o report
12. 看 HTML 报告分析结果
```

---

## 本节重点

- JMeter 核心树：测试计划 → 线程组 → 取样器/断言/定时器/监听器/配置元件
- 线程组三参数：**线程数、Ramp-Up、循环次数**（Ramp-Up 别设 0）
- 四必会：**参数化(CSV)、关联(提取器)、思考时间(定时器)、集合点(同步定时器)**
- 聚合报告看：**Error% → Throughput → 99% Line**
- 正式压测必须用**命令行非 GUI 模式**，`-n -t -l -e -o`
- 单机压不动用**分布式压测**（Master 调度 + 多 Slave 发压）
