## 第一部分：CAN 是什么，为什么用它

### 一句话

CAN（Controller Area Network，控制器局域网）是一种**车上各个 ECU 之间互相通信的总线**，1986 年博世（Bosch）为汽车设计的。

### 为什么车上用 CAN

早期汽车每两个部件之间要单独拉线，线束又粗又重又贵又难维护。

```text
没有总线：           有了 CAN 总线：
A ── B               A ─┐
A ── C                  │
B ── C   线爆炸       B ─┼── 一根总线 ──┤
A ── D                  │
...                  C ─┘
```

CAN 让所有 ECU 挂在**同一对线**上，谁都能发、谁都能收，线束大大简化。

### CAN 的物理特点

```text
- 双绞线：CAN_High 和 CAN_Low 两根线拧在一起（抗干扰）
- 差分信号：靠两根线的"电压差"传数据，抗干扰强
- 两端各一个 120Ω 终端电阻：防止信号反射
- 速率：经典 CAN 最高 1 Mbps
```

> 面试常问"终端电阻多少欧"——答 **120Ω，总线两端各一个**。

## 第二部分：CAN 的三大通信特点

### 特点1：多主（Multi-Master）

总线上没有"主机"，**任何节点都能主动发消息**，地位平等。谁想发就发，靠仲裁解决冲突。

### 特点2：广播（Broadcast）

一个节点发消息，**总线上所有节点都能收到**。收到后各自判断"这条跟我有没有关系"，有关就处理，无关就丢弃。

### 特点3：基于消息，不是基于地址

这是 CAN 最反直觉的一点：

```text
普通网络：消息里写"发给谁"（目标地址）
CAN：    消息里写"我是什么消息"（消息 ID），不写发给谁
```

CAN 帧里的 ID **不是地址，而是消息的身份/优先级**。比如 ID=0x100 代表"车速信号"，关心车速的节点自己来收。

> 这点面试爱问。记住：**CAN ID 标识的是消息内容和优先级，不是节点地址。**

## 第三部分：CAN 数据帧结构（必须吃透）

这是重点中的重点。一个标准 CAN 数据帧从头到尾的组成：

```text
┌──────┬────────┬──────┬────────┬─────┬─────┬──────┐
│帧起始│ 仲裁场  │控制场│ 数据场  │ CRC │ ACK │帧结束│
│ SOF  │  (ID)  │      │ 0~8字节 │     │     │ EOF  │
└──────┴────────┴──────┴────────┴─────┴─────┴──────┘
```

逐个解释：

| 字段 | 作用 |
|------|------|
| **帧起始 SOF** | 一个显性位，标志一帧开始 |
| **仲裁场** | 核心！包含 **ID**，决定优先级；还有 RTR 位 |
| **控制场** | 包含 DLC（数据长度码），说明数据有几个字节 |
| **数据场** | 真正的数据，**0~8 字节**（经典 CAN） |
| **CRC 场** | 循环冗余校验，检错用 |
| **ACK 场** | 接收方收到后发回确认 |
| **帧结束 EOF** | 标志一帧结束 |

### 标准帧 vs 扩展帧

```text
标准帧：ID 是 11 位  →  可表示 2^11 = 2048 个 ID
扩展帧：ID 是 29 位  →  可表示 2^29 ≈ 5.4 亿个 ID
```

ID 多了不够用，就用扩展帧。座舱里两种都可能见到。

## 第四部分：仲裁机制（理解优先级）

多个节点同时想发消息怎么办？CAN 用**非破坏性仲裁**解决——不会撞坏任何一条消息。

### 核心规则

```text
显性位（0）会"压过"隐性位（1）
ID 越小，优先级越高
```

### 怎么仲裁的

多个节点同时发，**逐位比较 ID**：

```text
节点A 的 ID： 0 0 1 1 0 1 ...
节点B 的 ID： 0 0 1 0 ...      ← 第4位 B 发0(显性)，A 发1(隐性)
                    ↑
              这一位 B 赢了（0 压过 1）
              A 检测到总线和自己发的不一样，立即退出
              B 继续发完，毫发无损
```

- 赢的节点（ID 小的）继续发，消息没受任何影响 → 所以叫"非破坏性"
- 输的节点自动退出，等下次空闲再发

> 一句话总结：**ID 数值越小 = 优先级越高 = 仲裁时越容易赢**。所以紧急消息（如刹车）会给小 ID。

## 第五部分：CAN FD

### CAN FD 是什么

CAN FD（Flexible Data-rate，灵活数据速率）是经典 CAN 的升级版，解决两个痛点：**数据太少、速度太慢**。

### 两大改进

```text
1. 数据场变长：经典 CAN 最多 8 字节  →  CAN FD 最多 64 字节
2. 可变速率：  仲裁阶段用低速（保证仲裁可靠）
              数据阶段切高速（传得快）
```

### 为什么需要它

车越来越智能，要传的数据越来越多。经典 CAN 一帧才 8 字节，传大数据要拆很多帧、很慢。CAN FD 一帧 64 字节，效率高得多。

> 记忆：**CAN FD = 更大的数据场（64字节）+ 数据段更快的速率**。

## 第六部分：错误处理（了解即可）

CAN 有一套自检错机制，知道这几个名词即可：

| 概念 | 说明 |
|------|------|
| **错误帧** | 节点检测到错误时，主动发一个错误帧通知全总线 |
| **错误计数器** | 每个节点记录自己收发的错误次数 |
| **错误主动 / 被动** | 错误多了，节点从"主动"降级为"被动"，发言权变小 |
| **总线关闭（Bus Off）** | 错误太多，节点彻底闭嘴，退出总线 |

> 这块座舱测试岗位不会深入，能说出"CAN 有错误帧和错误计数器，错误太多会进入 Bus Off 状态"就够了。

---

## 本节重点

- CAN 是车上 ECU 通信的总线，双绞线 + 差分信号 + **两端 120Ω 终端电阻**
- 三大特点：**多主、广播、基于消息（ID 不是地址）**
- 帧结构：SOF → 仲裁场(ID) → 控制场 → 数据场(0~8字节) → CRC → ACK → EOF
- 标准帧 11 位 ID，扩展帧 29 位 ID
- 仲裁：**ID 越小优先级越高，非破坏性仲裁**
- CAN FD：**数据场 64 字节 + 可变速率**
