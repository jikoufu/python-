# 消息队列基础

## 1. MQ 是什么

MQ 是 Message Queue，消息队列。它让系统之间通过消息通信，而不是直接同步调用。

简单理解：

```text
同步调用：A 直接调用 B，A 等 B 处理完。
消息队列：A 把消息发到 MQ，B 从 MQ 里慢慢消费。
```

## 2. MQ 解决什么问题

### 2.1 异步处理

用户注册后要发短信、发邮件、写日志。

不用 MQ：

```text
注册 -> 发短信 -> 发邮件 -> 写日志 -> 返回结果
```

用了 MQ：

```text
注册 -> 写入用户 -> 发送消息 -> 返回结果
消费者异步处理短信、邮件、日志
```

### 2.2 系统解耦

订单服务不需要直接调用库存、物流、优惠券、通知服务，只要发出订单创建消息。

### 2.3 削峰填谷

秒杀高峰时，请求先进入队列，消费者按能力处理，避免数据库瞬间被打爆。

## 3. MQ 带来的问题

引入 MQ 后，系统会更复杂：

| 问题 | 说明 |
|------|------|
| 消息丢失 | 生产、存储、消费任一环节都可能丢 |
| 重复消费 | 消息可能被消费者处理多次 |
| 消息积压 | 消费速度跟不上生产速度 |
| 顺序问题 | 消息消费顺序可能和发送顺序不同 |
| 一致性问题 | 主流程成功，异步任务失败 |

面试不要只说 MQ 好处，要主动补充这些风险。

## 4. 什么时候不适合用 MQ

- 业务必须立即得到下游结果。
- 系统规模很小，同步调用足够。
- 不能接受最终一致性。
- 团队没有能力维护 MQ 监控和补偿。

## 5. 面试回答模板

```text
MQ 主要解决异步、解耦、削峰问题。
比如订单创建后，可以通过 MQ 异步通知库存、物流和消息服务，主流程不用等待所有下游完成。
但 MQ 会带来消息丢失、重复消费、积压和最终一致性问题，所以需要 ACK、持久化、重试、死信队列、幂等和监控。
```
