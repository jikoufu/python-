# 消息可靠性与死信队列

## 1. 消息为什么会丢

消息链路：

```text
Producer -> RabbitMQ -> Queue -> Consumer
```

每一段都可能出问题：

| 环节 | 风险 |
|------|------|
| Producer -> RabbitMQ | 发送失败，网络断开 |
| RabbitMQ 存储 | MQ 宕机，消息没持久化 |
| RabbitMQ -> Consumer | 消费者处理失败但消息被确认 |

## 2. 如何保证消息不丢

### 2.1 生产者确认

Producer 发送消息后，RabbitMQ 确认收到，生产者才认为成功。

这叫 Publisher Confirm。

### 2.2 队列和消息持久化

队列持久化：

```text
durable=True
```

消息持久化：

```text
delivery_mode=PERSISTENT
```

### 2.3 消费者手动 ACK

消费者处理成功后再 ACK。

如果处理失败：

- NACK 后重新入队。
- 或进入死信队列。

## 3. 重复消费

即使做了可靠性保障，也可能重复消费。

原因：

```text
消费者处理成功，但 ACK 发送失败。
RabbitMQ 认为没处理成功，又把消息投递一次。
```

解决：消费者幂等。

常见方法：

- 用业务唯一 ID 去重。
- 数据库唯一索引。
- Redis 记录已处理消息 ID。
- 状态机控制，已完成状态不重复处理。

## 4. 消息积压

消息积压：队列中 Ready 消息越来越多，消费者处理不过来。

排查：

- 生产速度是否突然增加。
- 消费者是否挂了。
- 消费者处理是否变慢。
- 下游 MySQL/Redis/第三方服务是否慢。
- 是否有大量失败重试。

处理：

- 扩容消费者。
- 优化消费逻辑。
- 批量消费。
- 临时限流生产端。
- 失败消息进入死信队列。

## 5. 死信队列

死信队列 DLQ = Dead Letter Queue。

消息进入死信队列的常见原因：

- 消息过期。
- 消费失败并拒绝重新入队。
- 队列达到最大长度。
- 重试次数过多。

作用：

```text
保存异常消息，便于人工排查、重新投递或补偿。
```

## 6. 面试回答模板

```text
我会从三段保证消息可靠性：生产者开启发送确认，RabbitMQ 侧队列和消息都持久化，消费者侧手动 ACK。
同时消费者必须做幂等，因为消息可能重复投递。
如果消费失败，会通过重试和死信队列保留问题消息，避免消息直接丢失。
```

## 7. 测试点

| 场景 | 测试方式 |
|------|----------|
| 生产者发送失败 | 模拟 MQ 不可用，验证重试和日志 |
| 消费者处理失败 | 模拟异常，验证消息是否重试或进 DLQ |
| 重复消息 | 发送相同业务 ID，验证不会重复扣减 |
| 消息积压 | 降低消费者速度，观察队列 Ready 数 |
| MQ 重启 | 重启 RabbitMQ，验证持久化消息是否还在 |
| 消费者重启 | 消费中断后恢复，验证未 ACK 消息是否继续消费 |
