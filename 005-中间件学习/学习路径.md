## 总览：五阶段主线

```text
阶段1 MySQL  →  阶段2 Redis  →  阶段3 消息队列  →  阶段4 Nginx  →  阶段5 Nacos

         → 你要永远多问一句"这个特性怎么测？"
```

# 阶段 1：MySQL（必须最先且最深）🔥

## 测开为什么要会

几乎所有业务测试都要读写数据库：造测试数据、校验结果落库、验证数据一致性。而且索引/事务/锁的思维，是后面理解 Redis 缓存一致性、Kafka 消息一致性的**地基**。MySQL 是测开面试问得最深的中间件，没有之一。

## 必须掌握（面试清单）

```text
□ SQL 基础 + 多表 JOIN + 子查询
□ 索引：B+树、最左前缀、回表、索引失效场景  🔥
□ EXPLAIN 看执行计划，会判断有没有走索引  🔥
□ 事务 ACID + 四种隔离级别 + 各自解决/遗留什么问题  🔥
□ 锁：行锁/表锁、共享/排他锁、间隙锁、死锁
□ MVCC 原理（高级岗位 🔥）
□ 主从复制 + 主从延迟（测开高频，影响你读到的数据对不对）🔥
□ 慢查询定位与优化
```

## 是什么 / 为什么 / 怎么办 / 例子

### 索引（必考核心）

- **是什么**：加速查询的数据结构，MySQL(InnoDB) 用 **B+ 树**。
- **为什么用 B+树**：矮胖（3~4 层就能存千万数据，IO 少）、叶子节点链表相连（范围查询快）。
- **怎么办（最左前缀原则）**：联合索引 `(a,b,c)`，查询必须从最左 a 开始连续用，才能命中。
- **例子（索引失效，必考）**：

```sql
-- 联合索引 (name, age)
where name='张三' and age=20   -- ✅ 走索引
where age=20                    -- ❌ 跳过最左 name，失效
where name like '%三'          -- ❌ 左模糊，失效
where name='张三' or age=20    -- ❌ or 一侧无索引，失效
where left(name,1)='张'        -- ❌ 对字段用函数，失效
```

### 事务与隔离级别（必考核心）

- **是什么**：ACID（原子、一致、隔离、持久）。
- **为什么有隔离级别**：并发事务会互相干扰，产生三类问题：

```text
脏读     ：读到别人没提交的数据
不可重复读：同一事务两次读同一行，值变了（别人改了并提交）
幻读     ：同一事务两次查同范围，行数变了（别人插了新行）
```

- **怎么办（四级隔离，记这张表 🔥）**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| 读未提交 RU | ❌会 | ❌会 | ❌会 |
| 读已提交 RC | ✅防 | ❌会 | ❌会 |
| 可重复读 RR（MySQL默认）| ✅防 | ✅防 | ✅基本防(间隙锁) |
| 串行化 Serializable | ✅防 | ✅防 | ✅防 |

- **例子**：MySQL 默认 RR，靠 **MVCC + 间隙锁**在 RR 下基本解决了幻读，这是面试加分点。

### 主从延迟（测开高频 🔥）

- **是什么**：主库写、从库读，从库同步有延迟。
- **为什么测开必须知道**：你的自动化测试"写完马上读"，如果读走了从库，可能读到旧数据，**测试假性失败**。
- **怎么办**：理解原因（主从复制是异步的）；测试时强制读主库 / 加等待 / 用唯一标识轮询确认。

## 🔥 必考面试题（带答案要点）

```text
Q1: 为什么用 B+树不用 B树/哈希？
  → B+树矮胖IO少、叶子链表利于范围查；哈希不支持范围和排序。

Q2: 索引失效有哪些场景？
  → 不满足最左前缀、左模糊、字段用函数/运算、OR一侧无索引、类型隐式转换。

Q3: 四种隔离级别分别解决什么？MySQL 默认哪个？
  → 见上表；默认 RR，靠 MVCC+间隙锁基本解决幻读。

Q4: 什么是 MVCC？
  → 多版本并发控制，通过 undo log 版本链 + ReadView 实现读不加锁、读写不冲突。

Q5: 事务的隔离性靠什么实现？
  → 锁 + MVCC。

Q6: 主从延迟怎么产生？对测试有什么影响？怎么应对？
  → 异步复制导致；可能读到旧数据致测试误判；强制读主/轮询确认。
```

## 测开落地（apiforge）

```text
- 接口写入后，直连 DB 校验数据是否正确落库（不只信接口返回）
- 事务回滚验证：异常场景下数据有没有脏写
- 用 EXPLAIN 监控关键查询有没有走索引（性能回归）
```

# 阶段 2：Redis（紧跟数据库）🔥

## 测开为什么要会

缓存和 DB 是**一对**——缓存一致性问题本质是两者配合出错。学完 MySQL 再看缓存穿透/击穿/雪崩会非常顺。Redis 还常用于分布式锁、限流，都是测开要验证的点。

## 必须掌握（面试清单）

```text
□ 5 种基础数据结构 + 使用场景  🔥
□ 缓存穿透 / 击穿 / 雪崩 + 解决方案  🔥🔥（必考）
□ 缓存与DB一致性问题 + 双写策略  🔥
□ 持久化 RDB vs AOF
□ 过期策略 + 内存淘汰策略
□ 分布式锁（SETNX / Redisson）
□ 为什么这么快（内存 + 单线程 + IO多路复用）
```

## 是什么 / 为什么 / 怎么办 / 例子

### 三大缓存问题（必考中的必考 🔥🔥）

| 问题 | 是什么 | 为什么发生 | 怎么办 |
|------|--------|-----------|--------|
| **穿透** | 查根本不存在的数据，每次都打到 DB | 缓存里永远没有，挡不住 | 缓存空值 / 布隆过滤器 |
| **击穿** | 某个热点 key 失效瞬间，大量请求涌向 DB | 单个热 key 过期 | 互斥锁重建 / 热点 key 不过期 |
| **雪崩** | 大量 key 同时失效，DB 被打垮 | 过期时间集中 | 过期时间加随机值 / 多级缓存 / 熔断 |

- **例子（区分穿透 vs 击穿，面试爱问）**：

```text
穿透：查 id=-1 这种不存在的，缓存DB都没有 → "本来就没有"
击穿：查 id=1 这个爆款商品，缓存刚好过期 → "有但刚失效"
```

### 缓存与 DB 一致性（测开核心 🔥）

- **是什么**：DB 改了，缓存没跟着改，读到旧数据。
- **为什么难**：双写（写DB+写缓存）无法保证原子，且有并发。
- **怎么办（主流方案）**：**先更新 DB，再删除缓存**（Cache Aside）；删除而非更新，避免并发写脏；配合延迟双删 / 订阅 binlog 兜底。
- **例子（为什么是"删"不是"改"）**：两个请求并发改，"改缓存"可能后改的旧值覆盖新值；"删缓存"让下次读重新加载，更安全。

## 🔥 必考面试题

```text
Q1: 缓存穿透/击穿/雪崩区别和解决方案？（几乎必问）
  → 见上表。关键词：穿透=布隆过滤器/缓存空值，击穿=互斥锁，雪崩=过期加随机。

Q2: 怎么保证缓存和数据库一致性？
  → 先更DB再删缓存(Cache Aside)；延迟双删；binlog订阅兜底。
    并说明：强一致很难，一般做最终一致。

Q3: 为什么是删缓存而不是更新缓存？
  → 避免并发更新导致旧值覆盖新值；删除让下次读重建，更安全省资源。

Q4: RDB 和 AOF 区别？
  → RDB快照(快、可能丢数据)，AOF日志(全、文件大恢复慢)；可混合。

Q5: Redis 为什么快？单线程为什么不慢？
  → 纯内存 + IO多路复用 + 高效数据结构；单线程省去锁和上下文切换，瓶颈在内存/网络不在CPU。

Q6: Redis 怎么做分布式锁？有什么坑？
  → SET key val NX EX；坑：锁过期业务没执行完、误删别人的锁(用唯一value)、主从切换锁丢失(Redlock)。
```

## 测开落地（apiforge）

```text
- 缓存一致性校验：改完DB后读接口，验证拿到的是新值
- 缓存穿透防护验证：用不存在的id狂打，看DB有没有被击穿
- 分布式锁验证：并发请求下，互斥是否生效（库存不超卖）
```

# 阶段 3：消息队列 / Kafka（异步解耦削峰）🔥

## 测开为什么要会

异步、解耦、削峰是分布式系统核心场景，测开高频涉及**消息可靠性测试**。选 Kafka 是因为它的分区/副本/消费组模型最有代表性，**学透一个，RabbitMQ 等触类旁通**（本目录保留了 RabbitMQ 对比资料）。

## 必须掌握（面试清单）

```text
□ MQ 三大作用：解耦、异步、削峰  🔥
□ Kafka 核心概念：Topic/Partition/Replica/Producer/Consumer/ConsumerGroup  🔥
□ 消息不丢失（生产/存储/消费三端怎么保证）🔥🔥
□ 消息不重复 / 消费幂等  🔥🔥
□ 消息顺序性怎么保证  🔥
□ 消息积压怎么处理
□ Kafka vs RabbitMQ 选型区别
```

## 是什么 / 为什么 / 怎么办 / 例子

### Kafka 核心模型

```text
Topic(主题) 下分多个 Partition(分区) → 分区是并行和顺序的基本单位
每个 Partition 有多个 Replica(副本) → 保证高可用，一主多从
ConsumerGroup(消费组)：组内一个分区只被一个消费者消费 → 决定并行度
```

- **为什么分区**：提高并行度（多分区多消费者并行）；**为什么副本**：防止节点宕机丢数据。

### 消息可靠性"不丢不重不乱"（测开核心 🔥🔥）

**① 消息不丢失**——三端都要保证：

```text
生产端：acks=all（所有副本确认才算成功）+ 重试
存储端：副本数≥2，min.insync.replicas≥2
消费端：关闭自动提交，业务处理成功后再手动提交 offset
```

**② 消息不重复 / 幂等**：

```text
为什么会重复：网络重试、消费成功但提交offset失败 → 重复消费
怎么办：消费端做幂等。用唯一业务ID去重（DB唯一键/Redis记录已处理ID）
关键认知：MQ 无法绝对不重复，幂等要靠消费端兜底  🔥
```

**③ 消息顺序性**：

```text
Kafka 只保证「单分区内」有序，跨分区无序。
要顺序：把需要有序的消息(如同一订单)用相同key路由到同一分区。
```

- **例子（消费幂等）**：扣款消息重复消费，若不幂等会扣两次。解决：每条消息带唯一 txn_id，处理前查"这个 txn_id 处理过没"，处理过就跳过。

## 🔥 必考面试题

```text
Q1: 消息队列有什么用？什么场景用？
  → 解耦、异步、削峰。例:下单后发短信(异步)、秒杀流量削峰。

Q2: 怎么保证消息不丢失？（必考）
  → 生产acks=all+重试、存储多副本、消费手动提交offset。三端齐全。

Q3: 怎么保证消息不被重复消费？（必考）
  → MQ保证不了绝对不重复，靠消费端幂等：唯一ID去重。

Q4: 怎么保证消息顺序？
  → 同一类消息用相同key进同一分区；单分区有序。

Q5: 消息积压怎么办？
  → 临时扩消费者+扩分区；排查消费慢的原因；必要时先转储后处理。

Q6: Kafka 和 RabbitMQ 怎么选？
  → Kafka:高吞吐、日志/大数据、分区顺序；RabbitMQ:低延迟、复杂路由、业务消息。
```

## 测开落地（apiforge）

```text
- 不丢失验证：发N条消息，断言消费端最终收到N条
- 幂等验证：人为重复投递同一消息，断言业务只生效一次（如只扣一次款）
- 顺序验证：同key消息按序发送，断言消费顺序一致
- 积压观测：监控消费lag，验证削峰场景下不丢消息
```

# 阶段 4：Nginx（流量入口）⭐

## 测开为什么要会

测开常用 Nginx 搭测试环境、做反向代理、理解流量入口；面试常问反向代理和负载均衡。概念简单、上手快，当作轻松过渡。

## 必须掌握

```text
□ 正向代理 vs 反向代理 区别  🔥
□ 负载均衡 + 常见算法（轮询/加权/ip_hash/最少连接）🔥
□ 反向代理配置（location / proxy_pass）
□ 限流配置（限流测试要用）
□ 动静分离概念
```

## 是什么 / 为什么 / 怎么办 / 例子

### 正向 vs 反向代理（必考 🔥）

```text
正向代理：代理「客户端」，客户端知道目标，服务端不知真实客户端（如翻墙VPN）
反向代理：代理「服务端」，客户端以为在跟Nginx说话，不知背后有多少台服务器
口诀：正向藏客户端，反向藏服务端。
```

### 负载均衡算法

| 算法 | 怎么分 | 场景 |
|------|--------|------|
| 轮询(默认) | 依次分发 | 服务器性能相近 |
| 加权轮询 | 按权重分 | 服务器性能不均 |
| ip_hash | 同IP固定一台 | 需要会话保持 |
| 最少连接 | 给最闲的 | 请求耗时差异大 |

- **例子（ip_hash 解决什么）**：session 存在单机内存时，同一用户必须每次打到同一台，否则登录态丢失 → 用 ip_hash。

## 🔥 必考面试题

```text
Q1: 正向代理和反向代理区别？→ 正向藏客户端，反向藏服务端。
Q2: 负载均衡有哪些算法？ip_hash解决什么？→ 见表；会话保持。
Q3: Nginx为什么能扛高并发？→ 事件驱动+epoll，异步非阻塞，少量进程处理大量连接。
Q4: 怎么用Nginx做限流？→ limit_req(漏桶)、limit_conn(连接数)。
```

## 测开落地（apiforge）

```text
- 负载均衡验证：多次请求，断言被分发到不同后端
- 限流验证：超过阈值的请求被拒(429)，阈值内正常
```

# 阶段 5：Nacos（服务治理）⭐ 中高级

## 测开为什么要会

微服务场景才用得到，依赖你对"服务间如何通信"已有概念。中高级测开面试会问服务发现、配置中心。理解门槛建立在前面基础上，所以放最后。

## 必须掌握

```text
□ 注册中心是什么、解决什么问题  🔥
□ 服务注册与发现流程
□ 配置中心 + 配置热更新  🔥
□ 健康检查机制
□ (了解)CP vs AP，Nacos 的选择
```

## 是什么 / 为什么 / 怎么办 / 例子

### 注册中心（必考 🔥）

- **是什么**：服务的"通讯录"。服务启动时把自己的地址注册上去，调用方从这里查地址。
- **为什么需要**：微服务实例动态增减、IP 会变，不能在代码里写死地址。

```text
没有注册中心：服务A调服务B → 硬编码B的IP → B扩容/换IP就崩
有注册中心：  B启动→注册到Nacos；A→问Nacos要B的地址→调用
            B挂了→Nacos健康检查发现→剔除→A不会再调到坏的
```

### 配置中心 + 热更新

- **是什么**：集中管理所有服务的配置，改配置不用重启服务。
- **测开为什么关心**：要验证"改了配置，服务有没有实时生效"。

## 🔥 必考面试题

```text
Q1: 注册中心解决什么问题？→ 服务地址动态管理，解耦调用方和提供方。
Q2: 服务注册发现的流程？→ 提供方注册→消费方订阅/拉取→健康检查剔除故障节点。
Q3: 配置中心的好处？→ 集中管理+热更新，改配置不重启。
Q4: 注册中心挂了会怎样？→ 已缓存地址的调用仍可用一段时间(本地缓存)，新服务发现受影响。
```

## 测开落地（apiforge）

```text
- 服务上下线验证：停掉一个实例，验证流量不再打到它
- 配置热更新验证：改配置中心的值，验证服务行为实时变化（无需重启）
```
