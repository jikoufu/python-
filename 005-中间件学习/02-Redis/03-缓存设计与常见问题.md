# 缓存设计与常见问题

## 1. Cache Aside 模式

Cache Aside 也叫旁路缓存，是最常见的缓存模式。

### 读流程

```text
1. 先查 Redis。
2. Redis 有数据，直接返回。
3. Redis 没数据，查 MySQL。
4. 查到数据后写入 Redis。
5. 返回数据。
```

### 写流程

```text
1. 先更新 MySQL。
2. 再删除 Redis 缓存。
3. 下次读取时重新加载缓存。
```

为什么是删除缓存，而不是更新缓存：

```text
更新缓存可能因为并发导致旧数据覆盖新数据。
删除缓存更简单，下一次读再重建，可以降低一致性风险。
```

## 2. Python 示例

```python
import json

import redis.asyncio as redis

client = redis.from_url("redis://127.0.0.1:6379/0", decode_responses=True)


async def get_user(user_id: int):
    cache_key = f"user:{user_id}"

    cached = await client.get(cache_key)
    if cached:
        return json.loads(cached)

    user = await query_user_from_mysql(user_id)
    if user is None:
        await client.set(cache_key, json.dumps(None), ex=60)
        return None

    await client.set(cache_key, json.dumps(user), ex=3600)
    return user


async def update_user(user_id: int, data: dict):
    await update_user_in_mysql(user_id, data)
    await client.delete(f"user:{user_id}")
```

## 3. 缓存穿透

缓存穿透：查询一个不存在的数据，Redis 没有，MySQL 也没有，每次请求都会打到数据库。

例子：

```text
大量请求 user_id = -1 或不存在的 user_id。
```

解决方案：

- 缓存空值，但过期时间要短。
- 参数校验，非法 ID 直接拦截。
- 布隆过滤器，提前判断数据是否可能存在。

## 4. 缓存击穿

缓存击穿：一个热点 key 过期，大量请求同时访问，全部打到数据库。

例子：

```text
首页热门商品 key 过期，同一时间大量用户访问。
```

解决方案：

- 热点 key 不设置过短过期时间。
- 用互斥锁，只允许一个请求重建缓存。
- 后台定时刷新热点缓存。

## 5. 缓存雪崩

缓存雪崩：大量 key 同时过期，或者 Redis 整体不可用，导致请求直接打到数据库。

解决方案：

- 过期时间加随机值，避免同时过期。
- Redis 高可用。
- 本地缓存兜底。
- 接口限流和降级。

## 6. 缓存一致性

常见策略：

| 策略 | 说明 |
|------|------|
| 先更新 DB，再删除缓存 | 最常用 |
| 延迟双删 | 更新 DB 后删除缓存，延迟一段时间再删一次 |
| MQ 异步删除缓存 | 适合复杂系统 |
| 设置合理 TTL | 即使删除失败，也能最终过期 |

面试回答模板：

```text
我会优先采用 Cache Aside：读时先查缓存，未命中查 DB 并回写缓存；写时先更新 DB，再删除缓存。
同时设置 TTL 保证最终一致。如果并发较高，可以结合延迟双删或 MQ 做补偿。
```

## 7. 缓存测试点

| 测试点 | 关注内容 |
|--------|----------|
| 命中率 | 高频请求是否命中 Redis |
| 过期策略 | key 过期后是否正确重建 |
| 数据一致性 | 更新 DB 后缓存是否删除 |
| 异常兜底 | Redis 挂了接口是否能降级 |
| 热点 key | 热点失效是否压垮 DB |
| 并发读写 | 是否出现旧值覆盖新值 |
