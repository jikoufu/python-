# 中间件学习（测试开发视角）

> 这个目录不是给后端开发看的，是给**测试开发工程师**看的。
> 一切以**通过中级/高级测开面试**为目标——只学面试会考、工作会用的，不钻牛角尖。

## 一、测开为什么必须懂中间件

面试官问中间件，不是要你会运维，而是要确认三件事：

```text
1. 你懂系统是怎么搭起来的 → 才能设计出有深度的测试用例
2. 你知道哪里容易出问题   → 才能测到别人测不到的 bug（缓存一致性、消息丢失…）
3. 你能定位问题到中间件层 → 出了问题不只会说"接口报错"，能说清是哪一层
```

一句话：**中间件知识决定了你测试用例的深度，也决定了你定位问题的能力上限。** 这正是初级测试和中高级测开的分水岭。

## 二、学习主线与顺序

按"数据由内向外流动"的逻辑，也是依赖关系和面试频率的顺序：

```text
单机存储        加速层        解耦层         流量层        治理层
  MySQL    →    Redis   →   消息队列   →    Nginx   →    Nacos
 (必考最深)   (必考)      (Kafka,必考)   (高频)      (中高级)

   01           02           03            04           05
```

| 顺序 | 中间件 | 测开面试权重 | 一句话定位 |
|------|--------|------------|-----------|
| 01 | **MySQL** | ⭐⭐⭐⭐⭐ 必考最深 | 几乎所有业务测试的地基，索引/事务/锁是后面理解一致性的基础 |
| 02 | **Redis** | ⭐⭐⭐⭐⭐ 必考 | 缓存和 DB 是一对，缓存一致性是高频考点 |
| 03 | **消息队列(Kafka)** | ⭐⭐⭐⭐ 必考 | 异步解耦削峰，消息"不丢不重不乱序"是测开核心场景 |
| 04 | **Nginx** | ⭐⭐⭐ 高频 | 反向代理/负载均衡，搭测试环境、理解流量入口 |
| 05 | **Nacos** | ⭐⭐⭐ 中高级 | 微服务服务治理，中高级岗位才深入 |

### 两个"配角"的定位

```text
06-Docker        前置工具：测开天天用它搭环境/造数据/做CI，建议最先当工具学会，
                 但它不在"数据流主线"里，所以单列。面试也常问镜像/容器/compose。
07-Elasticsearch 进阶可选：日志检索、搜索类业务才深入，面试权重低于上面5个。
```

## 三、每篇文档怎么读

本目录学习文档统一用**测开面试导向**的结构：

```text
是什么   →  概念说清楚（能给面试官讲明白）
为什么   →  为什么需要它 / 为什么这么设计
怎么办   →  实际怎么用、怎么配置、怎么测
例子     →  具体场景和代码
常见面试题 →  高频会问
必考面试题 →  几乎一定考，给答案要点
测开落地  →  在自动化测试框架里写对应测试场景
```

## 四、学习方法（重要）

> **边学边在 apiforge 框架里落地。**

每学完一个中间件，立刻写对应的测试场景，比纯看原理记得牢得多：

```text
MySQL  → 写数据一致性校验、事务回滚验证、慢查询监控
Redis  → 写缓存一致性校验（DB改了缓存有没有跟着变）、缓存穿透防护验证
Kafka  → 写消息不丢/不重/顺序性验证、消费幂等性测试
Nginx  → 写负载均衡分发验证、限流生效验证
Nacos  → 写服务上下线、配置热更新验证
```

这些测试场景本身就是**面试作品集**和**项目经验**。
