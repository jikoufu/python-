# 中间件常见面试题

## 一、MySQL

### 1. 慢 SQL 怎么排查？

回答要点：

- 开启慢查询日志。
- 找到具体慢 SQL。
- 用 `EXPLAIN` 分析执行计划。
- 看是否走索引、扫描行数、是否 filesort/temporary。
- 检查是否返回数据太多、分页太深、关联表太大。
- 必要时加索引、改 SQL、拆查询、做缓存。

回答模板：

```text
我会先定位具体慢 SQL，再用 EXPLAIN 看 type、key、rows、Extra。
如果没有走索引，就检查 WHERE 条件和联合索引顺序。
如果走了索引但仍然慢，就看返回行数、排序、临时表、深分页或锁等待。
```

### 2. 什么情况下索引会失效？

- 对索引列使用函数。
- 左模糊查询。
- 隐式类型转换。
- 联合索引没有满足最左前缀。
- 使用 `OR` 时部分条件没有索引。
- 数据量太小或优化器判断全表扫描更便宜。

### 3. 主键索引和普通索引有什么区别？

InnoDB 中，主键索引是聚簇索引，叶子节点存整行数据。普通索引叶子节点存主键值，查到主键后可能还要回表查询整行数据。

### 4. 什么是回表？

通过普通索引查到主键后，再根据主键到主键索引中查整行数据，这个过程叫回表。

避免方式：

- 只查询索引中已有字段，形成覆盖索引。
- 合理设计联合索引。

### 5. 什么是幻读？

同一个事务内，两次范围查询看到的记录数量不一致。比如第一次查没有某条数据，另一个事务插入后，第二次查又出现了。

InnoDB 在 `REPEATABLE READ` 下通过 MVCC 和 Next-Key Lock 处理很多幻读问题。

### 6. 事务隔离级别怎么选？

普通互联网业务多数使用 MySQL 默认的 `REPEATABLE READ`。如果追求更接近已提交读，减少间隙锁影响，也有业务使用 `READ COMMITTED`。

面试不要只背表格，要说明权衡：

```text
隔离级别越高，一致性越强，但并发能力越低。
隔离级别越低，并发能力越高，但可能出现脏读、不可重复读、幻读。
```

## 二、Redis

### 1. Redis 常见数据结构和场景？

| 结构 | 场景 |
|------|------|
| String | 缓存、计数器、分布式锁 |
| Hash | 对象存储 |
| List | 队列、最新列表 |
| Set | 去重、共同关注 |
| ZSet | 排行榜、延迟任务 |

### 2. 缓存和数据库如何保持一致？

常见方案是 Cache Aside：

```text
读：先读缓存，缓存没有再读数据库，然后写缓存。
写：先更新数据库，再删除缓存。
```

为什么不是更新缓存？

```text
更新缓存容易引入并发覆盖问题，也可能更新了很少被读取的数据。
删除缓存更简单，下次读取时再重建。
```

### 3. Redis 持久化 RDB 和 AOF 区别？

| 方式 | 说明 | 优缺点 |
|------|------|--------|
| RDB | 定期生成快照 | 恢复快，可能丢最近数据 |
| AOF | 记录写命令 | 数据更完整，文件更大 |

生产环境常结合使用。

### 4. Redis 大 key 有什么问题？

问题：

- 占用内存大。
- 删除或读取可能阻塞。
- 网络传输慢。
- 集群分片不均。

处理：

- 拆分 key。
- 控制集合大小。
- 用扫描命令分批处理。
- 避免一次性返回大量数据。

### 5. Redis 热 key 怎么处理？

- 本地缓存。
- 多副本读。
- 热点 key 拆分。
- 限流。
- 提前预热。

## 三、消息队列

### 1. 如何处理重复消费？

消费者必须做幂等。

常见方式：

- 使用业务唯一 ID。
- 消费前查是否处理过。
- 数据库唯一索引防重复。
- Redis 记录处理状态。

回答模板：

```text
MQ 很难保证绝对只消费一次，所以消费者要按至少一次投递来设计。
我会通过业务唯一 ID、去重表或唯一索引保证重复消息不会造成重复扣款、重复发货这类问题。
```

### 2. 消息积压怎么办？

排查：

- 生产速度是否突然增加。
- 消费者是否异常。
- 单条消息处理是否变慢。
- 下游数据库或第三方服务是否变慢。

处理：

- 扩容消费者。
- 优化消费逻辑。
- 临时增加批量消费。
- 对失败消息进入死信队列。
- 必要时限流生产端。

### 3. 死信队列有什么用？

死信队列用于接收无法正常消费的消息，例如：

- 消息过期。
- 消费失败超过重试次数。
- 队列长度超限。
- 消费者主动拒绝且不重新入队。

作用是保留问题消息，便于人工排查和补偿。

## 四、Nginx

### 1. Nginx 和应用服务分别做什么？

Nginx：

- 静态资源。
- 反向代理。
- 负载均衡。
- HTTPS。
- 限流。
- 访问日志。

应用服务：

- 业务逻辑。
- 参数校验。
- 数据库操作。
- 权限控制。

### 2. 502、504 分别可能是什么问题？

| 状态码 | 常见原因 |
|--------|----------|
| 502 | 后端服务不可用、端口错、服务崩了 |
| 504 | 后端响应超时、接口慢、网络问题 |

排查：

- 看 Nginx error log。
- 检查 upstream 配置。
- 检查后端服务是否启动。
- 检查应用日志和接口耗时。

### 3. Nginx 如何限流？

常见配置：

```nginx
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

location /api/ {
    limit_req zone=api burst=20 nodelay;
    proxy_pass http://backend;
}
```

含义：按客户端 IP 限制请求速率。

## 五、Docker

### 1. Dockerfile 中 COPY 和 ADD 区别？

`COPY` 只复制文件。`ADD` 除了复制，还支持自动解压 tar 和远程 URL。

实际项目优先用 `COPY`，语义更明确。

### 2. docker compose 解决什么问题？

解决多容器编排问题。例如一个后端项目同时需要：

- FastAPI。
- MySQL。
- Redis。
- RabbitMQ。
- Nginx。

用 compose 可以一条命令启动全部服务：

```bash
docker compose up -d
```

### 3. 容器删除后数据为什么没了？

因为容器内部文件系统是临时的。删除容器后，容器内写入的数据也会消失。

MySQL、Redis、ES 这类有状态服务必须使用 volume：

```yaml
volumes:
  mysql_data:
```

## 六、Elasticsearch

### 1. ES 为什么适合全文搜索？

因为 ES 使用倒排索引。它不是按文档找词，而是按词找文档，所以适合关键词检索、模糊匹配、相关性排序。

### 2. ES 和 MySQL 查询有什么区别？

| 项目 | MySQL | Elasticsearch |
|------|-------|---------------|
| 核心能力 | 事务、关系查询 | 全文搜索、聚合分析 |
| 索引结构 | B+Tree 为主 | 倒排索引 |
| 数据一致性 | 强一致更好 | 近实时 |
| 典型场景 | 订单、用户、交易 | 搜索、日志、内容检索 |

### 3. MySQL 和 ES 数据怎么同步？

常见方案：

- 业务双写。
- MQ 异步同步。
- 监听 MySQL binlog，例如 Canal。

测试关注：

- MySQL 更新后 ES 是否同步。
- 同步失败是否重试。
- 删除数据是否同步删除。
- ES 查询结果是否存在延迟。

## 七、综合排查题

### 1. 接口返回数据不是最新的，怎么查？

可能原因：

- Redis 缓存未删除。
- 数据库事务未提交。
- 读写分离读到了从库延迟数据。
- ES 同步延迟。
- 前端或网关缓存。

排查顺序：

```text
接口返回值 -> 应用日志 -> Redis 缓存 -> MySQL 主库 -> 从库/ES -> Nginx/前端缓存
```

### 2. 发布后接口大量报错，怎么查？

先看错误类型：

- 连接数据库失败：MySQL 配置、账号、网络、连接池。
- Redis 失败：缓存服务、连接配置、超时。
- MQ 失败：队列、交换机、消费者。
- 502/504：Nginx upstream 或后端服务。
- 数据不一致：事务、缓存、异步消息。

面试表达：

```text
我会先按错误码和日志定位是应用层问题还是中间件问题。
如果是中间件，会继续确认连接、配置、容量、慢查询、队列积压、缓存命中率等指标。
最后把问题归因到配置、代码、容量或数据四类，再推动修复和回归。
```
