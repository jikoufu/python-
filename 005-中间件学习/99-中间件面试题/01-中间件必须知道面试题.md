# 中间件必须知道面试题

这些题属于面试高频基础题。你如果面 AI 质量工程、测试开发、后端测试岗位，至少要能清楚回答。

## 1. 什么是中间件？

中间件是业务系统和底层基础设施之间的通用组件，用来解决存储、缓存、通信、代理、搜索、日志等问题。

常见中间件：

| 类型 | 例子 | 解决的问题 |
|------|------|------------|
| 数据库 | MySQL | 持久化存储 |
| 缓存 | Redis | 提升读性能、分布式锁、计数 |
| 消息队列 | RabbitMQ、Kafka | 异步解耦、削峰填谷 |
| 网关/代理 | Nginx | 反向代理、负载均衡、限流 |
| 搜索 | Elasticsearch | 全文搜索、日志检索 |
| 容器 | Docker | 环境隔离、部署编排 |

回答模板：

```text
中间件是业务系统中可复用的基础组件，不直接承载核心业务，但支撑业务运行。
比如 MySQL 负责数据持久化，Redis 负责缓存和分布式锁，MQ 负责异步解耦，Nginx 负责代理和负载均衡。
测试时我会关注它们的配置、数据一致性、性能瓶颈、异常恢复和链路可观测性。
```

## 2. MySQL 索引为什么能提升查询速度？

因为索引减少了扫描数据的范围。InnoDB 常用 B+Tree 索引，可以通过树结构快速定位数据。

必须补充：

- 索引不是越多越好。
- 索引会占空间。
- 写入、更新、删除时需要维护索引，会影响写性能。
- 联合索引要遵循最左前缀。

回答模板：

```text
索引本质是减少扫描行数。没有索引可能全表扫描，有索引可以通过 B+Tree 快速定位。
但索引也有成本，写入时要维护索引，所以要根据高频查询条件设计。
我一般会用 EXPLAIN 看 type、key、rows、Extra，确认是否真正用上索引。
```

## 3. MySQL 事务是什么？ACID 怎么理解？

事务是一组数据库操作，要么全部成功，要么全部失败。

| 特性 | 说明 |
|------|------|
| 原子性 | 全成功或全失败 |
| 一致性 | 事务前后数据符合约束和业务规则 |
| 隔离性 | 并发事务互不干扰 |
| 持久性 | 提交后数据持久保存 |

项目回答示例：

```text
比如转账场景，一个账户扣款，另一个账户加款，这两个操作必须在同一个事务里。
如果扣款成功但加款失败，就要回滚，否则数据不一致。
```

## 4. Redis 缓存穿透、击穿、雪崩分别是什么？

| 问题 | 现象 | 解决 |
|------|------|------|
| 缓存穿透 | 查不存在的数据，请求都打到数据库 | 缓存空值、布隆过滤器 |
| 缓存击穿 | 热点 key 过期，大量请求同时查数据库 | 分布式锁、热点 key 续期 |
| 缓存雪崩 | 大量 key 同时过期或 Redis 挂了 | 过期时间加随机值、多级缓存、限流降级 |

回答模板：

```text
穿透是查不存在的数据，击穿是热点 key 失效，雪崩是大量 key 同时失效或缓存整体不可用。
解决思路分别是缓存空值/布隆过滤器、热点 key 加锁重建、过期时间随机化和降级限流。
```

## 5. Redis 为什么快？

主要原因：

- 数据存内存。
- 单线程处理命令，避免频繁线程切换和锁竞争。
- I/O 多路复用处理网络连接。
- 数据结构设计高效。

注意：Redis 不是所有操作都快。大 key、慢命令、阻塞操作会拖慢 Redis。

## 6. Redis 分布式锁怎么实现？

基础做法：

```text
SET lock_key unique_value NX EX 10
```

解释：

- `NX`：不存在才设置，保证互斥。
- `EX`：设置过期时间，避免死锁。
- `unique_value`：释放锁时校验是不是自己加的锁。
- 释放锁要用 Lua 脚本保证“判断 + 删除”的原子性。

回答模板：

```text
我会用 SET key value NX EX ttl 加锁，value 用唯一 token。
释放时用 Lua 脚本判断 token 一致再删除，避免误删别人的锁。
同时要考虑锁超时、业务执行过长、失败重试和幂等问题。
```

## 7. 消息队列解决什么问题？

核心价值：

- 异步处理：注册后发短信、发邮件不用阻塞主流程。
- 系统解耦：订单服务不直接依赖库存、物流、通知服务。
- 削峰填谷：流量高峰先进入队列，消费者慢慢处理。

回答模板：

```text
MQ 主要解决异步、解耦和削峰。
但引入 MQ 后也会带来消息丢失、重复消费、顺序消费、积压等问题，所以需要 ACK、持久化、幂等、死信队列和监控。
```

## 8. 如何保证 MQ 消息不丢？

从三个环节回答：

| 环节 | 保障方式 |
|------|----------|
| 生产者到 MQ | 发送确认、失败重试 |
| MQ 自身 | 队列持久化、消息持久化 |
| MQ 到消费者 | 手动 ACK、消费失败重试、死信队列 |

补充：即使保证不丢，也可能重复消费，所以消费者必须做幂等。

## 9. Nginx 反向代理和负载均衡是什么？

反向代理：客户端访问 Nginx，由 Nginx 转发到后端服务，客户端不直接访问真实服务。

负载均衡：多个后端实例同时提供服务，Nginx 按策略分发请求。

常见策略：

- 轮询。
- 权重。
- 最少连接。
- IP hash。

## 10. Docker 镜像和容器区别？

| 概念 | 解释 |
|------|------|
| 镜像 | 只读模板，类似安装包 |
| 容器 | 镜像运行后的实例，类似进程 |
| 数据卷 | 保存容器外部数据 |
| 网络 | 容器之间通信 |

回答模板：

```text
镜像是静态模板，容器是镜像运行后的实例。
容器删除后内部数据可能丢失，所以 MySQL、Redis 这类有状态服务需要挂载数据卷。
```

## 11. 线上接口慢，你怎么判断是不是中间件问题？

排查路径：

1. 看接口耗时分布，是所有接口慢还是某几个接口慢。
2. 看应用日志，定位慢在哪个阶段。
3. 如果是数据库，查慢 SQL、连接池、锁等待。
4. 如果是 Redis，查大 key、慢命令、连接数、命中率。
5. 如果是 MQ，查队列积压、消费者处理耗时、失败重试。
6. 如果是 Nginx，查 upstream 响应时间、连接数、限流情况。

回答模板：

```text
我会先从链路耗时拆分，不直接猜。
通过日志或 trace 看耗时发生在应用、MySQL、Redis、MQ 还是 Nginx。
然后针对具体中间件看慢 SQL、缓存命中率、队列积压、连接池、锁等待等指标。
```

## 12. 测试岗位为什么要懂中间件？

因为很多问题不是接口代码本身的问题，而是中间件引起的：

- 数据没写进去：可能是事务回滚。
- 接口偶发慢：可能是慢 SQL 或连接池耗尽。
- 数据不一致：可能是缓存和数据库不同步。
- 任务没执行：可能是 MQ 消息积压或消费失败。
- 发布后访问失败：可能是 Nginx 转发配置错误。

面试表达：

```text
测试开发需要理解中间件，不是为了替代 DBA 或运维，而是为了在质量保障中快速定位问题边界。
我至少要能判断问题出在业务代码、数据库、缓存、消息队列还是网关配置，并推动对应同学闭环。
```
