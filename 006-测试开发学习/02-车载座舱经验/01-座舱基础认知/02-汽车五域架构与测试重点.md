# 02 汽车五域架构与测试重点

## 学习目标

学完这一节，你需要掌握：

- 什么是汽车五域架构。
- 五个域分别负责什么。
- 车载座舱属于哪个域。
- 座舱测试为什么必须理解其他域。
- 五域架构下测试应该从哪些方面考虑。

## 什么是五域架构

汽车电子电气架构从早期“很多 ECU 分散控制”，逐渐向“域控制器集中控制”演进。

五域架构就是把整车功能大致划分成五个域：

```text
动力域
底盘域
车身域
座舱域
智驾域
```

每个域有自己的控制器、功能范围和通信链路。

简单理解：

```text
动力域：让车跑起来
底盘域：让车开得稳
车身域：管理车身电器
座舱域：负责人机交互
智驾域：负责感知、决策和辅助驾驶
```

## 五域总览

```text
整车电子电气架构
├── 动力域
├── 底盘域
├── 车身域
├── 座舱域
└── 智驾域
```

车载座舱测试主要在座舱域，但很多功能会和车身域、动力域、底盘域、智驾域联动。

所以不能只看中控屏页面。

## 1. 动力域

动力域负责车辆动力相关功能。

燃油车里通常和发动机、变速箱相关。

新能源车里通常和电机、电池、电控相关。

常见内容：

- 发动机
- 电机
- 电池
- 电控
- 变速箱
- 能量回收
- 充电状态
- 剩余续航

和座舱相关的测试点：

- 中控显示电量是否正确。
- 仪表显示续航是否正确。
- 充电状态是否同步。
- 能量回收模式切换是否显示正确。
- 动力告警是否在仪表/中控正确显示。

示例：

```text
动力域上报电量 20%
    ↓
仪表应显示 20%
    ↓
中控车辆状态页也应显示 20%
```

## 2. 底盘域

底盘域负责车辆行驶稳定和控制。

常见内容：

- 制动
- 转向
- 悬架
- ABS
- ESP
- 驻车
- 胎压
- 驾驶模式

和座舱相关的测试点：

- 驾驶模式切换 UI 是否正确。
- 胎压告警是否显示。
- 驻车状态是否同步。
- 制动告警是否显示。
- 悬架模式变化是否展示正确。

示例：

```text
底盘域上报胎压异常
    ↓
仪表出现胎压告警
    ↓
中控车辆状态页显示具体轮胎异常
```

## 3. 车身域

车身域负责车身电子电器。

这是座舱测试最常联动的域之一。

常见内容：

- 车门
- 车窗
- 后备箱
- 车灯
- 雨刮
- 后视镜
- 空调相关执行
- 座椅
- 门锁
- 氛围灯

和座舱相关的测试点：

- 中控车控页面操作是否发出正确信号。
- 车门状态变化后 UI 是否同步。
- 车窗开度显示是否正确。
- 灯光开关状态是否一致。
- 座椅加热/通风是否反馈正确。
- 行驶中是否限制危险车控。

示例：

```text
用户在中控点击打开车窗
    ↓
座舱域发出请求
    ↓
车身域执行车窗控制
    ↓
车身域反馈车窗状态
    ↓
中控 UI 更新车窗状态
```

## 4. 座舱域

座舱域负责人机交互和信息娱乐。

这是我们学习车载座舱经验的核心。

常见内容：

- IVI 中控
- 仪表
- HUD
- 副驾屏
- 后排屏
- 导航
- 多媒体
- 蓝牙电话
- 语音助手
- 空调 UI
- 车辆设置
- 应用生态

座舱域测试重点：

- 页面显示
- HMI 交互
- 多屏联动
- 语音交互
- 导航和地图
- 多媒体播放
- 蓝牙电话
- 车辆状态显示
- 和其他域的信号一致性

示例：

```text
中控开始导航
    ↓
仪表显示导航箭头
    ↓
HUD 显示距离和方向
    ↓
语音播报道路信息
```

## 5. 智驾域

智驾域负责辅助驾驶和自动驾驶相关能力。

常见内容：

- 摄像头
- 毫米波雷达
- 激光雷达
- 感知
- 目标识别
- 车道线识别
- ACC 自适应巡航
- LKA 车道保持
- AEB 自动紧急制动
- APA 自动泊车

和座舱相关的测试点：

- 智驾状态是否在仪表显示。
- ADAS 告警是否及时展示。
- 自动泊车页面是否正确。
- 智驾接管提示是否明确。
- 语音/提示音/视觉提示是否一致。

示例：

```text
智驾域触发接管提醒
    ↓
仪表显示接管提示
    ↓
座舱播放提示音
    ↓
HUD 显示警示信息
```

## 五域和座舱测试的关系

座舱域不是孤立的。

它经常接收其他域的信号，并把结果展示给用户。

也可能从用户操作出发，向其他域发送控制请求。

典型链路：

```text
其他域信号 → 座舱显示
座舱操作 → 其他域执行
其他域反馈 → 座舱状态更新
```

比如：

```text
车身域车门状态 → 中控/仪表显示车门
底盘域胎压异常 → 仪表/中控告警
动力域电量变化 → 仪表/中控续航显示
智驾域接管提醒 → 仪表/HUD/声音提示
座舱点击空调 → 车身或空调控制器执行 → 反馈状态
```

## 五域架构下的测试思路

测试时可以按这几个维度考虑：

### 1. 信号来源

这个 UI 状态来自哪个域？

例如：

```text
车门状态来自车身域
电量来自动力域
胎压来自底盘域
接管提醒来自智驾域
```

### 2. 信号链路

信号从哪里来，经过哪里，到哪里显示？

例如：

```text
车身域 → 网关 → 座舱域控 → 中控 UI
```

### 3. 显示一致性

多个屏幕是否显示一致？

例如：

```text
中控显示车门打开
仪表也应显示车门打开
HUD 如有相关提示也应一致
```

### 4. 操作闭环

用户操作是否真正执行，并反馈回来？

例如：

```text
点击开启座椅加热
    ↓
发出请求信号
    ↓
执行器反馈开启
    ↓
UI 显示开启
```

不能只看 UI 变了就算成功。

### 5. 安全限制

某些操作需要根据车速、档位、车辆状态限制。

例如：

- 行驶中禁止视频播放。
- 行驶中限制复杂设置。
- 非 P 档禁止部分车控。
- 车速过高时禁止危险操作。

### 6. 异常恢复

要测试：

- 信号丢失怎么办。
- 反馈超时怎么办。
- 下电上电后状态怎么恢复。
- 某个域重启后座舱显示是否恢复。

## 五域测试用例示例

### 用例 1：车门状态显示

```markdown
## 用例编号

DOMAIN_BODY_001

## 用例标题

车身域上报左前门打开后，中控和仪表显示一致

## 前置条件

- 车机启动完成。
- CANoe 工程已连接。
- DBC 加载正确。

## 测试步骤

1. 使用 CANoe 模拟左前门状态为打开。
2. 查看中控车辆状态页面。
3. 查看仪表车门状态显示。
4. 使用 CANoe 模拟左前门状态为关闭。
5. 再次查看中控和仪表显示。

## 预期结果

- 左前门打开时，中控和仪表均显示左前门打开。
- 左前门关闭时，中控和仪表均恢复关闭。
- 状态变化无明显延迟。
```

### 用例 2：车速限制视频播放

```markdown
## 用例编号

DOMAIN_CHASSIS_001

## 用例标题

车速大于阈值时，座舱限制视频播放

## 前置条件

- 视频功能可用。
- CANoe 可模拟车速信号。

## 测试步骤

1. 设置车速为 0。
2. 打开视频页面并播放。
3. 设置车速为 30 km/h。
4. 观察视频播放状态和提示。

## 预期结果

- 静止状态允许播放。
- 行驶状态限制播放。
- 提示文案符合需求。
```

### 用例 3：智驾接管提醒

```markdown
## 用例编号

DOMAIN_ADAS_001

## 用例标题

智驾域触发接管提醒后，仪表和声音提示正确

## 前置条件

- 智驾模拟环境可用。
- 仪表和音频输出正常。

## 测试步骤

1. 模拟智驾接管提醒信号。
2. 查看仪表是否显示接管提示。
3. 检查是否播放提示音。
4. 信号恢复后查看提示是否消失。

## 预期结果

- 接管提示显示及时。
- 声音提示正确。
- 信号恢复后提示消失。
```

## 面试表达

如果面试官问：

```text
你了解汽车五域架构吗？
```

可以这样回答：

```text
了解。常见五域包括动力域、底盘域、车身域、座舱域和智驾域。座舱测试主要在座舱域，但很多功能依赖其他域的信号，比如车门、车窗、座椅来自车身域，车速、胎压、驾驶模式可能和底盘域相关，电量和续航来自动力域，接管提醒和辅助驾驶状态来自智驾域。所以做座舱测试不能只看 UI，还要结合 CANoe 或信号工具验证信号来源、显示一致性和操作闭环。
```

如果继续问：

```text
五域架构对座舱测试有什么影响？
```

可以回答：

```text
影响主要在跨域联动和信号一致性。座舱页面展示的很多状态不是自己产生的，而是来自其他域。测试时要确认信号来源是否正确、UI 是否及时刷新、多屏显示是否一致，以及用户在座舱发起的车控操作是否被其他域执行并反馈。尤其是车控类功能，不能只看 UI 状态，要看请求信号和反馈信号。
```

## 本节重点

- 五域架构包括动力域、底盘域、车身域、座舱域、智驾域。
- 座舱域负责中控、仪表、HUD、语音、导航、多媒体等。
- 座舱测试必须理解其他域，因为很多 UI 状态来自其他域信号。
- 车控测试要看操作闭环：请求、执行、反馈、显示。
- CANoe 是验证跨域信号和座舱显示一致性的常用工具。
