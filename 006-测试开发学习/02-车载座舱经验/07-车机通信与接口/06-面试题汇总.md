## 一、CAN 基础

### ⭐ Q1：简单介绍一下 CAN 总线？

**参考答案**：CAN 是博世为汽车设计的一种总线，让车上各个 ECU 挂在同一对双绞线上互相通信，大大简化了线束。它有三个特点：多主（任何节点都能主动发）、广播（所有节点都能收到）、基于消息（消息里写的是 ID 标识消息内容和优先级，不是目标地址）。物理上用差分信号抗干扰，总线两端各有一个 120Ω 终端电阻防反射。

**要点**：多主 + 广播 + 基于消息 + 120Ω 终端电阻。

### ⭐ Q2：CAN 的 ID 是地址吗？

**参考答案**：不是。这是 CAN 最特别的地方——CAN 是"基于消息"而不是"基于地址"。ID 标识的是这条消息的内容和优先级，比如 0x100 代表车速信号。节点发消息时不指定发给谁，所有节点都收到后自己判断要不要处理。同时 ID 还决定优先级，越小优先级越高。

**要点**：ID = 消息身份 + 优先级，不是地址。

### Q3：CAN 总线的终端电阻是多少？为什么要有？

**参考答案**：120Ω，总线两端各一个。作用是阻抗匹配，防止信号在总线末端反射造成干扰。

---

## 二、CAN 帧结构与仲裁

### ⭐ Q4：CAN 数据帧由哪些部分组成？

**参考答案**：依次是帧起始(SOF)、仲裁场(含 ID)、控制场(含数据长度 DLC)、数据场(经典 CAN 0~8 字节)、CRC 校验场、ACK 确认场、帧结束(EOF)。其中仲裁场的 ID 最关键，既标识消息又决定优先级。

**要点**：能按顺序背出来，重点突出仲裁场和数据场。

### ⭐ Q5：两个节点同时发送，会冲突吗？怎么解决？

**参考答案**：不会撞坏，CAN 用非破坏性仲裁。多个节点同时发时逐位比较 ID，显性位(0)会压过隐性位(1)，所以 ID 数值越小的越容易赢。赢的节点继续发，消息完好无损；输的节点检测到总线和自己发的不一致，立即退出，等下次再发。所以叫"非破坏性"——优先级高的消息一点不受影响。

**要点**：非破坏性仲裁 + ID 越小优先级越高 + 显性压隐性。

### Q6：标准帧和扩展帧的区别？

**参考答案**：标准帧 ID 是 11 位，能表示约 2000 个 ID；扩展帧 ID 是 29 位，能表示约 5.4 亿个，ID 不够用时用扩展帧。

## 三、CAN FD

### ⭐ Q7：CAN FD 和经典 CAN 有什么区别？

**参考答案**：两大改进。第一，数据场变长，经典 CAN 一帧最多 8 字节，CAN FD 最多 64 字节。第二，可变速率，仲裁阶段用低速保证可靠，数据阶段切到高速传得快。本质是为了应对车上数据越来越多、经典 CAN 带宽不够的问题。

**要点**：64 字节 + 可变速率。

---

## 四、LIN

### Q8：LIN 和 CAN 什么关系？用在哪？

**参考答案**：LIN 是 CAN 的低成本补充。它是主从架构（一个主节点调度多个从节点），单线通信，速率低（约 20kbps），成本低。用在对实时性要求不高的低速场景，比如车窗、座椅调节、灯光、后视镜这些。

**要点**：主从 + 低速低成本 + CAN 的补充 + 车窗座椅场景。

---

## 五、车载以太网（座舱重点）

### ⭐ Q9：座舱为什么大量用以太网，而不是 CAN？

**参考答案**：座舱要传的是摄像头图像、高清音视频、高精地图、域控之间的大量数据，这些都是大带宽数据。CAN 一帧才 8 字节，CAN FD 也才 64 字节，根本扛不住。以太网带宽大（百兆千兆起步），适合传这种大数据。

**要点**：大带宽数据（图像/音视频/地图）→ CAN 扛不住 → 用以太网。

### ⭐ Q10：车载以太网和普通以太网有什么区别？

**参考答案**：最大区别在物理层。普通以太网用 8 芯网线（4 对双绞线），车载以太网用单对双绞线（就 2 根线），标准是 100BASE-T1（百兆）和 1000BASE-T1（千兆），T1 就是指单对双绞线。这样设计是为了省线束重量和成本。上层协议栈（IP、TCP/UDP）和普通以太网基本一样。

**要点**：单对双绞线 + 100BASE-T1/1000BASE-T1 + 省线束。

### ⭐ Q11：知道 SOME/IP 吗？

**参考答案**：SOME/IP 是车载以太网上面向服务的通信协议。和 CAN 的"面向信号"（不停广播信号）不同，它是"面向服务"，按需通信。三个核心概念：服务发现（节点广播自己能提供什么服务，别人来发现）、方法调用（客户端请求、服务端处理返回，一问一答）、事件订阅（订阅后服务端在数据变化时主动推送）。

**要点**：面向服务 + 服务发现、方法调用、事件订阅。

### Q12：DoIP 和 TSN 是什么？

**参考答案**：DoIP 是基于以太网的诊断，比传统走 CAN 的诊断快得多，刷写读码效率高。TSN 是时间敏感网络，给以太网加实时性保证，让音视频同步这类对时间敏感的数据能准时到达。

## 六、网关与整车通信

### ⭐ Q13：网关在整车里起什么作用？

**参考答案**：整车有 CAN、LIN、以太网多种总线，它们协议不同，网关就是中枢，负责在不同总线之间转换协议、路由数据。比如刹车信号在动力域 CAN 上，座舱要显示刹车图标，这个信号就要经网关从 CAN 转到座舱以太网。座舱里很多信号其实来自其他域，都是网关转过来的。

**要点**：跨总线协议转换 + 路由 + 举座舱例子。

### Q14：说一下整车通信的数据链路？

**参考答案**：域内 ECU 之间用 CAN/LIN 通信，域控制器之间走以太网骨干，跨域时由网关做协议转换和路由。比如传感器数据 → 域内 CAN → 域控制器 → 以太网骨干 → 网关 → 座舱域 → 最后显示在中控屏/仪表上。

## 七、AUTOSAR

### ⭐ Q15：AUTOSAR 是什么？为什么需要它？

**参考答案**：AUTOSAR 是汽车软件的标准化架构规范。以前软件和硬件死绑，换芯片就要大改，不同供应商代码不通用。AUTOSAR 解决三件事：软硬件解耦、标准化、复用。通过分层和标准接口，让上层软件不关心底层硬件，换硬件只改底层。

**要点**：解耦 + 标准化 + 复用。

### ⭐ Q16：CP 和 AP 有什么区别？座舱用哪个？

**参考答案**：CP（Classic Platform）面向传统实时控制类 ECU，比如发动机、车身控制，用实时操作系统，面向信号通信，算力小。AP（Adaptive Platform）面向高性能智能 ECU，基于 POSIX 操作系统（类似 Linux/QNX），面向服务通信（SOME/IP），算力大，支持 OTA 动态升级。简单说，CP 管实时控制类 ECU，AP 管算力大的智能化 ECU。**座舱属于 AP**，因为它需要大算力跑复杂应用。

**要点**：CP 实时控制 / AP 智能高算力，座舱是 AP。

### Q17：CP 的分层架构？

**参考答案**：从上到下是应用层、RTE（运行时环境，上下层之间的接口）、BSW（基础软件，提供通信诊断 OS 等服务）、MCAL（微控制器抽象层，直接操作芯片）。核心思想是分层封装，换芯片只改 MCAL，应用层不动。

## 八、DBC 与实际工作

### ⭐ Q18：DBC 文件是什么？

**参考答案**：DBC 是 CAN 报文的"翻译字典"，定义了每条报文（Message）有哪些信号（Signal），每个信号的起始位、长度、字节序、缩放、偏移等。有了它才能把总线上的原始字节翻译成有意义的物理值，比如把 0x1234 算成 46.6 km/h。

### ⭐ Q19：信号怎么从原始字节算成实际值？

**参考答案**：用公式"物理值 = 原始值 × 缩放(factor) + 偏移(offset)"。先根据起始位、长度、字节序从字节里取出原始值，再套这个公式算出物理值。

### ⭐ Q20（结合 Python 的加分题）：你怎么解析 CAN 报文？

**参考答案**：我用 Python 的 cantools 库。加载 DBC 后，用 decode_message 就能把原始字节直接解码成信号字典，比如 `{'VehicleSpeed': 46.6}`；要发报文就用 encode_message 把物理值打包成字节。纯软件、不需要硬件就能跑。实际工作中我会用它来核对总线信号——比如座舱屏幕车速显示不对时，我先解析总线上的真实车速信号，如果总线值是对的，说明问题在座舱软件而不是总线，这样能快速划清责任边界。

**要点**：这题把 Python 优势和实际排查能力结合起来，是你的差异化亮点，一定要讲好。

## 九、综合 / 软实力题

### Q21：总线出问题，你作为测试怎么配合定位？

**参考答案**：我会先用工具（CANoe 或 cantools 脚本）抓总线报文，对照 DBC 解析出相关信号的实际值，判断问题出在哪个环节：如果总线信号本身就错了，说明是上游 ECU 或网关的问题，我会把抓包数据提供给总线/对应域的工程师；如果总线信号是对的、但座舱表现不对，那就是座舱软件的问题。核心是用数据定位，划清责任边界，而不是凭感觉甩锅。

### Q22：你对总线的掌握到什么程度？

**参考答案**（诚实且得体）：我的定位是能讲清原理、看懂报文、配合定位问题。CAN 的帧结构、仲裁、车载以太网和 SOME/IP 这些原理我能讲明白，DBC 报文解析我能用 Python 实际操作。深度的总线一致性测试、网络管理开发不是我的主攻方向，但我能和总线工程师顺畅对话、配合工作。

**要点**：这题考自我认知。**别吹自己是总线专家**，如实说"原理清楚、能配合"反而显得靠谱。

## 高频必考题速记（面试前过一遍）

1. CAN 三特点：多主、广播、基于消息
2. 终端电阻：120Ω，两端各一
3. CAN 帧结构：SOF→仲裁(ID)→控制→数据(0~8)→CRC→ACK→EOF
4. 仲裁：非破坏性，ID 越小优先级越高
5. CAN FD：64 字节 + 可变速率
6. 车载以太网：单对双绞线，100BASE-T1
7. SOME/IP：服务发现、方法调用、事件订阅
8. 网关：跨总线转换路由
9. CP vs AP：实时控制 vs 智能高算力，座舱是 AP
10. DBC + cantools：物理值 = 原始值 × 缩放 + 偏移
