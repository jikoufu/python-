# 08 测试配置、日志与报告

## 是什么

测试配置用于统一 pytest 的运行规则；日志用于记录测试过程；报告用于展示测试结果。它们让测试从“能跑”变成“能交付”。

常见配置文件：

- `pytest.ini`
- `pyproject.toml`
- `tox.ini`

## 为什么要用它

当测试项目变大后，需要统一管理：

- 默认测试路径。
- mark 注册。
- 日志级别。
- 报告输出。
- 覆盖率统计。
- CI 需要的 JUnit XML。

没有统一配置，团队成员运行结果可能不一致。

## 怎么用

`pytest.ini` 示例：

```ini
[pytest]
testpaths = tests
python_files = test_*.py
python_functions = test_*
addopts = -ra -q
markers =
    smoke: smoke tests
    regression: regression tests
```

生成 HTML 报告：

```bash
pip install pytest-html
pytest --html=reports/report.html --self-contained-html
```

生成覆盖率报告：

```bash
pip install pytest-cov
pytest --cov=src --cov-report=term-missing --cov-report=html
```

生成 CI 常用 XML：

```bash
pytest --junitxml=reports/junit.xml
```

## 场景例子

场景：每次运行测试都默认生成报告。

```ini
[pytest]
testpaths = tests
addopts =
    -ra
    --html=reports/report.html
    --self-contained-html
    --junitxml=reports/junit.xml
```

这样本地和 CI 运行时都有一致的结果文件。

## 大厂使用的例子

大型团队通常会把测试报告接入平台：

- CI 页面展示测试通过率。
- 覆盖率报告用于质量门禁。
- 失败日志和截图作为缺陷定位材料。
- 历史趋势用于观察质量变化。

例如一个接口自动化项目，失败时会保存请求参数、响应体、环境、日志和报告链接，方便开发快速复现。

## 本章小结

配置、日志和报告决定了自动化测试能不能被团队真正使用。测试结果不只是控制台上的通过或失败，还要能沉淀、追踪和复盘。
