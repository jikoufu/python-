# 12 自动化测试框架封装

## 是什么

自动化测试框架封装，是把 pytest、请求客户端、配置、日志、断言、数据、报告组织成一套可维护的工程结构。

它不是简单把代码放进很多目录，而是让团队能稳定新增、运行、维护测试。

## 为什么要用它

零散脚本的问题：

- 接口地址写死。
- 登录逻辑重复。
- 断言分散。
- 测试数据混在代码里。
- 报告和日志不可追踪。
- 新人不知道该把用例写在哪里。

框架封装的目标是降低重复和维护成本。

HTTP Client 示例：

```python
import requests


class HttpClient:
    def __init__(self, base_url, headers=None):
        self.base_url = base_url
        self.headers = headers or {}

    def get(self, path, **kwargs):
        return requests.get(f"{self.base_url}{path}", headers=self.headers, **kwargs)

    def post(self, path, json=None, **kwargs):
        return requests.post(
            f"{self.base_url}{path}",
            json=json,
            headers=self.headers,
            **kwargs,
        )
```

## 场景例子

场景：封装登录后的接口客户端。

```python
@pytest.fixture(scope="session")
def api_client(base_url):
    token = login_and_get_token(base_url)
    return HttpClient(
        base_url=base_url,
        headers={"Authorization": f"Bearer {token}"},
    )


def test_get_profile(api_client):
    response = api_client.get("/profile")
    assert response.status_code == 200
```

## 大厂使用的例子

大型团队常见框架能力：

- 多环境配置切换。
- 多服务客户端封装。
- 统一鉴权和 token 刷新。
- 数据驱动。
- 公共业务断言。
- 失败自动保存请求和响应。
- 报告自动上传测试平台。

例如微服务团队会按服务封装 client：`UserClient`、`OrderClient`、`PaymentClient`，测试用例只表达业务流程，不重复关心底层 HTTP 细节。

## 本章小结

框架封装要服务于维护效率，不要为了“看起来高级”过度设计。先让用例清晰、数据独立、配置统一、报告可查，再逐步抽象公共能力。
