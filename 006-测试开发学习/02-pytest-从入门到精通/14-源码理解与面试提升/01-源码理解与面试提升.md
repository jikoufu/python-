# 14 源码理解与面试提升

## 是什么

源码理解不是要求一开始就读完整 pytest 源码，而是理解 pytest 的核心机制：

- 测试发现
- 用例收集
- fixture 依赖解析
- 参数化生成用例
- mark 标记和筛选
- Hook 插件机制
- 报告生成过程

## 为什么要用它

当你只会写用例时，只能解决常规问题。理解机制后，才能处理：

- fixture 为什么执行多次或只执行一次。
- 参数化为什么生成多条用例。
- `conftest.py` 为什么不用 import。
- mark 为什么要注册。
- 插件为什么能影响测试过程。
- CI 里为什么本地能过、线上不过。

这些也是测试开发面试高频问题。

## 怎么用

建议按机制而不是按源码文件死读：

1. 先理解测试发现规则。
2. 再理解 fixture 生命周期。
3. 再理解参数化如何生成多条测试。
4. 再理解 Hook 插件机制。
5. 最后结合真实问题看源码。

可以用命令观察 pytest 行为：

```bash
pytest --collect-only
pytest -vv
pytest --fixtures
pytest --markers
```

查看可用 fixture：

```bash
pytest --fixtures
```

只收集不执行：

```bash
pytest --collect-only
```

## 场景例子

场景：解释 fixture 执行顺序。

```python
import pytest


@pytest.fixture
def db():
    print("connect db")
    yield "db"
    print("close db")


def test_user(db):
    assert db == "db"
```

理解点：

- 测试函数声明了 `db` 参数。
- pytest 根据参数名找到同名 fixture。
- 执行测试前先执行 `yield` 前逻辑。
- 测试结束后执行 `yield` 后逻辑。

## 大厂使用的例子

大型团队面试或晋升时，通常不只看“会不会写用例”，还会看：

- 能否设计自动化测试框架。
- 能否解释 pytest 核心机制。
- 能否定位 flaky 用例。
- 能否把测试接入 CI 并治理稳定性。
- 能否基于 Hook 做内部平台集成。

例如质量平台要采集每条用例的模块、负责人、耗时和失败截图，就需要理解 pytest 执行生命周期和报告 Hook。

## 本章小结

源码理解的最终目标不是背代码，而是把 pytest 的运行机制讲清楚，并能用这些机制解决工程问题。能从“会用”走到“会解释、会扩展、会落地”，才算真正进阶。
