# 01 滴滴 AI 质量工程师面试准备

## 岗位判断

从 JD 看，这不是普通手工测试岗位，也不是纯算法模型评测岗位。

它更像：

```text
测试开发 / 质量工程 + AI 质量工具落地 + Agent Workflow / Prompt / Dataset / 评测平台建设
```

面试重点大概率会覆盖三类：

```text
1. 传统测试开发基本功
2. AI 大模型应用测试能力
3. AI 赋能测试效率的落地能力
```

## 你要先准备的核心答案

### 1. Agent 自动化框架你了解吗？

建议回答：

```text
了解一些。Agent 自动化框架本质上是把 LLM、工具调用、状态管理、记忆、工作流编排和执行轨迹管理组合起来，让模型可以完成多步骤任务。常见框架包括 LangGraph、AutoGen、CrewAI。测试视角不会只看最终回答，还要看 workflow 每一步是否正确，比如工具选择、参数、执行顺序、失败恢复、权限边界和执行 trace。
```

补充：

```text
如果要落到测试提效，可以设计测试用例生成 Agent、接口自动化脚本生成 Agent、缺陷归因 Agent、测试报告生成 Agent。工程上需要控制输入输出格式、保存中间轨迹、支持人工确认、加入规则校验和回归评测。
```

### 2. 你有没有做过 AI 质量工具？

如果没有实际做过，不要硬编。

建议回答：

```text
我之前更多是做传统测试开发和 AI 测试学习沉淀，没有完整上线过团队级 AI 质量平台。但我理解 AI 质量工具可以从低风险场景落地，比如需求生成测试点、接口用例生成、自动化脚本生成、缺陷日志分析、测试报告生成。关键不是只调用模型，而是要有 prompt 模板、输入规范、输出 schema、人工确认、失败样本沉淀和回归评测。
```

### 3. 你怎么理解 Agent Workflow？

回答结构：

```text
Agent Workflow = 多步骤 AI 工作流。
它通常包含任务拆解、工具调用、状态流转、结果校验、失败重试和人工确认。
```

测试视角：

```text
我会关注每个节点输入输出是否正确、工具调用是否符合预期、参数是否正确、失败是否能恢复、是否有越权调用、trace 是否可观测，以及最终结果是否满足业务要求。
```

### 4. 评测样本怎么建设？

回答：

```text
评测样本不是随便攒问题，而是要按业务场景、风险等级和失败类型分类建设。比如 Prompt 类样本关注格式、角色、安全；RAG 类样本关注检索命中、答案忠实性、引用准确；Agent 类样本关注工具选择、参数、步骤和权限。每条样本要包含输入、期望结果、评估规则、标签、优先级和历史失败信息。
```

## JD 知识点标注

### 业务日常质量保障

需要准备：

- 需求评审怎么做
- 测试点怎么拆
- 测试用例怎么设计
- 联调验收怎么做
- 回归测试怎么做
- 发布前风险怎么判断
- 线上问题怎么跟进闭环

可能被问：

```text
给你一个新需求，你怎么保障质量？
需求评审你会关注什么？
上线前你怎么判断能不能发？
线上问题你怎么定位和复盘？
```

准备答案要包含：

```text
需求理解 → 风险识别 → 用例设计 → 联调验收 → 回归策略 → 发布检查 → 线上监控 → 问题复盘
```

### AI 质量工具

需要准备：

- LLM 能帮测试做什么
- 测试用例生成
- 自动化脚本生成
- 测试数据生成
- 缺陷分析
- 日志归因
- 测试报告生成

可能被问：

```text
你会怎么设计一个 AI 测试用例生成工具？
如何保证 AI 生成的用例质量？
AI 生成脚本怎么避免不可靠？
```

回答重点：

```text
Prompt 模板 + 结构化输出 + 规则校验 + 人工确认 + 失败样本回流 + 持续评测
```

### Agent Workflow

需要准备：

- Agent 是什么
- Workflow 是什么
- Tool Call
- Memory
- Trace
- State
- Human-in-the-loop
- 失败重试
- 权限控制

可能被问：

```text
Agent 和普通 LLM 调用有什么区别？
Agent Workflow 怎么测试？
Tool Call 怎么验证？
怎么防止 Agent 调错工具？
```

回答重点：

```text
不能只看最终答案，要看中间步骤和工具调用轨迹。
```

### 评测样本 Dataset

需要准备：

- Dataset 字段设计
- 样本分类
- 标签体系
- 标注质量
- golden answer
- regression cases
- 高风险样本

可能被问：

```text
你怎么构建 AI 评测集？
样本从哪里来？
如何保证样本质量？
失败样本怎么处理？
```

回答重点：

```text
线上真实问题 + 需求高频问题 + 历史缺陷 + 红队攻击 + 边界样本 + 人工标注 + 版本管理
```

### 测试沙盒

需要准备：

- 什么是测试沙盒
- 隔离环境
- 模拟服务
- Mock 数据
- 工具调用隔离
- 权限隔离
- 安全验证

可能被问：

```text
为什么 Agent 测试需要沙盒？
测试沙盒要隔离什么？
Agent 调工具会有什么风险？
```

回答重点：

```text
Agent 会执行动作，所以测试环境必须隔离真实数据和真实操作，防止误删、误发、越权调用。
```

### 质量平台

需要准备：

- 用例管理
- 评测集管理
- 执行记录
- 评测报告
- 模型/Prompt 版本
- 失败样本沉淀
- CI 集成

可能被问：

```text
如果让你做一个 AI 质量平台，你会设计哪些模块？
怎么做 Prompt 回归？
怎么追踪模型版本变化带来的质量波动？
```

回答可以按模块说：

```text
样本管理、执行引擎、评估器、报告中心、Prompt 版本、模型版本、失败样本库、CI 入口、权限管理。
```

## 你可以主打的项目表达

如果没有真实 AI 质量平台经历，可以准备一个“方案型项目”。

项目名：

```text
AI 测试提效与评测样本管理工具
```

项目目标：

```text
利用 LLM 辅助生成测试用例、自动化脚本、缺陷分析和测试报告，并通过样本集和规则校验控制输出质量。
```

核心模块：

```text
1. Prompt 模板管理
2. Dataset 样本管理
3. 用例生成 Agent
4. 脚本生成 Agent
5. 缺陷分析 Agent
6. 评估器
7. 测试报告生成
8. 失败样本回流
```

测试闭环：

```text
输入需求 → LLM 生成测试点 → 规则校验 → 人工确认 → 生成用例 → 生成脚本 → 执行测试 → 失败分析 → 报告输出 → 样本沉淀
```

风险控制：

```text
结构化输出
规则断言
人工确认
沙盒执行
权限隔离
日志 trace
失败样本回归
```

## 面试底线

不要说自己“完整做过 AI 质量平台”，除非确实做过。

更稳的表达：

```text
我有测试开发和质量保障基础，也系统学习并沉淀了 AI 测试方向，包括 Prompt、RAG、Agent、评测样本和自动化评测。实际落地上，我可以从小范围专项切入，比如用例生成、脚本生成、缺陷分析或 RAG 评测集建设，然后逐步沉淀为团队可复用工具。
```

这个表达符合 JD 的“独立推动小范围专项落地”。
