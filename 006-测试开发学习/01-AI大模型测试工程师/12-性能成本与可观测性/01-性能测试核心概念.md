# 01 性能测试核心概念

## 学习目标

学完这一节，你需要掌握：

- 性能测试是什么、为什么要做
- 性能测试的几种类型（负载、压力、稳定性、并发、容量）
- 核心指标：TPS、RT、并发数、吞吐量、错误率
- 这些指标之间的关系
- 怎么定一个合理的性能目标

---

## 一、性能测试是什么

功能测试回答"功能对不对"，性能测试回答"系统**快不快、扛不扛得住、稳不稳**"。

```text
功能测试：1 个用户操作，结果对不对？
性能测试：1000 个用户同时操作，系统还撑得住吗？响应还快吗？会不会崩？
```

### 为什么要做性能测试

- 上线前知道系统能扛多少用户（容量评估）
- 找出系统的瓶颈，提前优化
- 验证大促/高峰场景下系统不崩
- 给扩容、配置提供数据依据

---

## 二、性能测试的类型（面试常问）

| 类型 | 目的 | 通俗说法 |
|------|------|---------|
| **负载测试** | 在预期负载下看性能表现 | "正常上班高峰扛得住吗" |
| **压力测试（压测）** | 不断加压直到系统崩，找上限 | "往死里压，看什么时候崩" |
| **并发测试** | 多用户同时操作同一功能 | "1000 人同时抢一张票" |
| **稳定性测试（疲劳测试）** | 长时间运行看会不会出问题 | "连续跑 7×24 小时会不会内存泄漏" |
| **容量测试** | 找出系统最大处理能力 | "最多能支撑多少用户" |
| **基准测试** | 单用户/小负载下的基础性能 | "没人抢的时候有多快" |

> 面试高频："负载测试和压力测试有什么区别？"
> 答：负载测试看**预期负载下表现是否达标**；压力测试是**持续加压找崩溃点和上限**。

---

## 三、核心指标（必须吃透）

这几个指标是性能测试的语言，面试必问，工作天天用。

### 1. 响应时间 RT（Response Time）

一个请求从发出到收到响应的耗时。

```text
RT = 网络传输 + 服务端处理 + 排队等待 ...
```

注意**不要只看平均值**，要看分位值：

```text
平均 RT：    所有请求耗时的平均（会被掩盖问题）
TP90 / P90： 90% 的请求在这个时间内完成
TP99 / P99： 99% 的请求在这个时间内完成（更能反映用户体验）
```

> 为什么看 P99 不看平均？因为平均值会被"大多数快的"拉低，掩盖掉"少数慢得离谱"的请求。P99 才暴露真实的尾部延迟。

### 2. TPS（Transactions Per Second）

每秒处理的事务数，**衡量系统吞吐能力的核心指标**。

```text
TPS = 总事务数 / 总时间
```

一个"事务"可以是一个接口调用，也可以是一组操作（如"登录→下单→支付"算一个事务），由你定义。

### 3. QPS（Queries Per Second）

每秒查询数。和 TPS 类似，但通常 QPS 指单个接口/查询，TPS 指业务事务。

```text
一个下单事务(TPS=1) 可能包含 多个接口查询(QPS=N)
```

### 4. 并发数（Concurrency）

同一时刻**正在处理**的请求数 / 同时在线施压的用户数。

> 注意区分：**并发用户数 ≠ 在线用户数**。100 万人在线，但同一秒真正在点按钮的可能只有几千。

### 5. 吞吐量（Throughput）

单位时间内系统处理的数据量，可以用 TPS 衡量，也可以用每秒字节数（KB/s）衡量。

### 6. 错误率（Error Rate）

```text
错误率 = 失败请求数 / 总请求数
```

压测时错误率突然飙升，往往就是系统到达瓶颈的信号。

## 四、指标之间的关系（重要规律）

这是理解性能测试的关键，面试也爱考。

### 黄金公式

```text
并发数 ≈ TPS × 平均响应时间(RT)
```

（即利特尔法则 Little's Law）

直观理解：TPS 越高、RT 越长，系统里同时"在跑"的请求就越多。

### 加压过程的三个阶段

随着并发数不断增加，系统会经历：

```text
TPS
 ↑
 │           ┌──────  ← ③ 拐点后：TPS 不升反降，RT 飙升，开始报错（崩溃区）
 │         ╱
 │       ╱  ← ② 拐点：TPS 达到最大值（系统上限）
 │     ╱
 │   ╱  ← ① 线性增长：加压 TPS 就涨，RT 平稳（健康区）
 │ ╱
 └─────────────────────→ 并发数
```

- **阶段①（线性增长区）**：加并发，TPS 线性上升，RT 基本不变 → 系统健康
- **阶段②（拐点）**：TPS 达到峰值 → 这就是系统的**最大处理能力**
- **阶段③（崩溃区）**：再加压，TPS 反而下降，RT 暴涨，错误率飙升 → 系统过载

> **性能测试的核心目标，就是找到那个"拐点"**——系统的真实上限在哪。

## 五、怎么定性能目标

不要凭感觉说"系统要很快"，要有量化目标。一般这样定：

```text
1. 估算业务量：日活、峰值 QPS（常用"二八原则"：80% 流量集中在 20% 时间）
2. 算峰值 TPS：日总请求 × 80% / (24h × 20% × 3600s)
3. 定指标门槛：
   - 峰值 TPS ≥ 目标值
   - P99 响应时间 ≤ X 毫秒
   - 错误率 ≤ 0.1%
   - CPU / 内存使用率 ≤ 80%（留余量）
```

举例目标描述：

> "在 500 并发下，下单接口 TPS ≥ 800，P99 响应时间 ≤ 200ms，错误率 ≤ 0.1%，服务器 CPU ≤ 75%。"

这样的目标才**可测、可验收**。

---

## 本节重点

- 性能测试回答"快不快、扛不扛得住、稳不稳"
- 类型：负载、压力、并发、稳定性、容量、基准（重点分清负载 vs 压力）
- 核心指标：**RT（看 P99 不看平均）、TPS、并发数、吞吐量、错误率**
- 黄金公式：**并发数 ≈ TPS × RT**
- 加压三阶段：线性增长 → 拐点（上限）→ 崩溃
- 性能目标必须量化、可验收
