# 03 性能瓶颈分析

> **这一章是性能测试的灵魂。** 会用 JMeter 发压的人遍地都是，但能拿着压测结果说清"瓶颈在哪、为什么、怎么解决"的人才值钱。面试时这块答得好，直接拉开档次。

## 学习目标

学完这一节，你需要能：

- 建立一套定位瓶颈的系统方法（分层排查）
- 掌握 CPU、内存、磁盘 IO、网络的分析手段
- 定位数据库、连接池、锁、GC 这类高频瓶颈
- 看懂关键监控指标，知道什么数值算异常
- 把"现象 → 定位 → 原因 → 优化"讲成一条完整链路

---

## 一、分析的总思路：自上而下分层排查

性能瓶颈不能瞎猜，要按层排查。一个请求会经过这么多层，瓶颈可能在任何一层：

```text
客户端(发压机)
    ↓
网络 / 带宽
    ↓
负载均衡 (Nginx/LB)
    ↓
应用服务 (CPU / 内存 / 线程 / GC / 连接池)
    ↓
中间件 (缓存 Redis / 消息队列 MQ)
    ↓
数据库 (慢查询 / 锁 / 连接数)
    ↓
磁盘 IO / 存储
```

### 排查铁律

```text
1. 先确认不是发压机自己的问题（客户端 CPU/网卡打满会让结果失真）
2. 看现象：TPS 上不去？RT 高？错误率高？现象不同方向不同
3. 自上而下逐层看监控，找资源被打满或异常的那一层
4. 定位到层后，深入找根因
5. 优化后再压一遍验证（一次只改一个变量）
```

---

## 二、先看现象，定方向

不同现象指向不同的排查方向：

| 现象 | 可能方向 |
|------|---------|
| TPS 上不去，但 CPU/内存都不高 | 有锁、串行瓶颈、线程池/连接池太小、外部依赖慢 |
| TPS 上不去，CPU 打满 | 计算密集、代码效率低、GC 频繁、序列化开销 |
| RT 越来越长 | 排队、数据库慢、连接池等待、GC 停顿、内存泄漏 |
| 错误率突然飙升 | 到达系统上限、超时、连接池耗尽、线程拒绝 |
| 压一会就崩 / RT 持续涨 | 内存泄漏、连接泄漏、句柄耗尽 |
| TPS 抖动剧烈、周期性卡顿 | Full GC、定时任务、缓存集中失效 |

> **"TPS 上不去但资源不高"是最经典的面试场景**——这往往不是机器不够，而是有锁、连接池/线程池配置太小，或卡在某个慢的外部依赖上。记住这个反直觉点。

---

## 三、逐项分析

### 1. CPU

```text
看什么：CPU 使用率、us(用户态)、sy(内核态)、wa(IO等待)、load average
工具：top / vmstat / mpstat
```

| 现象 | 含义 |
|------|------|
| us 高 | 应用代码计算密集（算法差、循环多、序列化重） |
| sy 高 | 内核态忙（频繁系统调用、上下文切换多） |
| wa 高 | 在等 IO（磁盘/网络），CPU 空着干等 → 瓶颈其实在 IO |
| load 远高于核数 | 系统过载，大量任务在排队 |

定位代码热点：Java 用 `top -Hp <pid>` 找到耗 CPU 的线程，转成 16 进制在 `jstack` 里查是哪段代码；或用 Arthas、火焰图。

### 2. 内存

```text
看什么：内存使用率、是否持续上涨不回落（泄漏信号）
Java 还要看：堆使用、GC 频率和耗时、是否频繁 Full GC
工具：free / jstat / jmap / 火焰图
```

经典内存问题：

- **内存泄漏**：内存只涨不降，最后 OOM。压测跑久了 RT 持续变差就要怀疑
- **频繁 Full GC**：每次 Full GC 会"Stop The World"暂停应用，表现为 TPS 周期性掉零、RT 尖刺
  - 看 `jstat -gcutil <pid> 1000`，Full GC 次数频繁增长 = 有问题
  - 原因：堆太小、对象创建过多、大对象、内存泄漏

### 3. 磁盘 IO

```text
看什么：IOPS、读写吞吐、IO 等待、磁盘使用率(%util)
工具：iostat -x 1
```

- `%util` 接近 100% = 磁盘忙不过来
- `await`（IO 平均等待时间）高 = IO 慢
- 常见原因：大量日志写入、数据库无索引全表扫描、临时文件

### 4. 网络

```text
看什么：带宽是否打满、连接数、TIME_WAIT 堆积、丢包重传
工具：iftop / netstat / ss / sar -n
```

- 带宽打满：传输大数据（图片/文件）时常见
- 大量 TIME_WAIT：短连接太多，考虑长连接/连接复用
- 注意：**发压机网卡打满**会让你误以为是服务端瓶颈

---

## 四、数据库瓶颈（最高频）

实际工作中，**性能瓶颈一大半在数据库**。重点排查：

### 1. 慢查询

```text
- 开慢查询日志，找出执行慢的 SQL
- EXPLAIN 分析执行计划：有没有走索引、是不是全表扫描(type=ALL)
- 常见根因：缺索引、索引失效、SELECT *、深分页 limit 100000,10
```

### 2. 数据库连接数 / 连接池

```text
- 连接池太小：应用大量请求排队等连接 → TPS 上不去但 DB 不忙
- 连接池太大：DB 连接数被打满，DB 压力过大
- 连接泄漏：用完不释放，连接越来越少最后耗尽
```

### 3. 锁

```text
- 行锁/表锁竞争：大量请求改同一行数据，串行等待 → 高并发下 TPS 卡死
- 死锁：互相等待，请求超时
- 典型场景：秒杀扣库存，所有请求抢同一行 → 必须用乐观锁/Redis 预扣等优化
```

> 面试场景："下单接口压测 TPS 很低，数据库 CPU 也不高，怎么回事？"
> 答：很可能是**行锁竞争**（都在抢同一行库存）或**连接池太小**导致请求排队，而不是 DB 算不过来。

## 五、应用层瓶颈

### 1. 线程池 / 连接池配置不当

```text
线程池太小：请求排队，TPS 上不去
线程池太大：上下文切换开销大，反而变慢
Tomcat 最大线程数、DB 连接池、HTTP 客户端连接池都要看
```

### 2. 同步阻塞 / 串行调用

```text
一个请求里串行调好几个外部服务，耗时叠加
优化：能并行的并行调、能异步的异步、加缓存减少调用
```

### 3. 缓存问题

```text
缓存击穿：热点 key 失效瞬间，大量请求直接打到 DB
缓存雪崩：大量 key 同时失效，DB 被打垮（表现为周期性 TPS 暴跌）
缓存穿透：查不存在的数据，每次都打 DB
```

## 六、瓶颈定位的完整案例（面试可直接套）

把方法落成一个完整故事：

```text
【现象】
订单接口压测，并发加到 200 时 TPS 停在 300 不再涨，
继续加并发 RT 从 100ms 涨到 2s，但应用服务器 CPU 才 40%。

【排查】
1. 先确认发压机正常（CPU、网卡都不高）→ 排除客户端问题
2. CPU 不高但 TPS 上不去 → 怀疑锁或等待，不是算力问题
3. 看应用线程栈 jstack → 大量线程 BLOCKED 在等数据库连接
4. 看 DB 连接池 → 最大连接 20，已全部占满，请求在排队
5. 看慢查询 → 订单查询 SQL 没走索引，单条 80ms，占着连接不放

【根因】
慢 SQL 拖长了单请求占用连接的时间 + 连接池太小，双重作用导致请求排队。

【优化】
1. 给订单查询字段加索引，SQL 从 80ms 降到 5ms
2. 连接池从 20 调到 50
3. 重新压测：TPS 从 300 提升到 1500，RT 回到 80ms

【结论】
瓶颈不在 CPU，在"慢 SQL + 连接池过小"导致的连接等待。
```

> 这个"现象→排查→根因→优化→验证"的结构，是面试讲性能案例的黄金模板。**一定要有数据、有前后对比。**

---

## 七、常用监控指标速查

| 层 | 关键指标 | 异常信号 |
|----|---------|---------|
| CPU | 使用率、load、wa | 使用率>80%、wa 高、load>核数 |
| 内存 | 使用率、GC 频率/耗时 | 只涨不降、频繁 Full GC |
| 磁盘 | %util、await | %util≈100%、await 高 |
| 网络 | 带宽、TIME_WAIT | 带宽打满、TIME_WAIT 堆积 |
| DB | 慢查询、连接数、锁等待 | 慢 SQL、连接打满、锁等待多 |
| 应用 | 线程池、连接池、RT | 线程阻塞、连接池耗尽 |

---

## 本节重点

- 瓶颈分析靠**自上而下分层排查**，不靠猜
- 先看现象定方向：**"TPS 上不去但资源不高" = 锁/连接池/串行等待**（最经典）
- CPU 看 us/sy/wa；内存重点防泄漏和 Full GC；DB 是头号瓶颈
- 数据库三查：**慢查询、连接池、锁**
- 面试讲案例用黄金模板：**现象 → 排查 → 根因 → 优化 → 数据验证**
