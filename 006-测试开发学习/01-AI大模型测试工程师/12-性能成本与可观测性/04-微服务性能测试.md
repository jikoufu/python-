# 04 微服务性能测试

## 学习目标

学完这一节，你需要能：

- 说清微服务架构给性能测试带来了哪些新挑战
- 区分单服务压测、链路压测、全链路压测
- 定位微服务特有的瓶颈（服务间调用、网关、注册中心、熔断限流）
- 理解分布式追踪在性能分析里的作用
- 知道什么是全链路压测、生产压测怎么做

---

## 一、微服务为什么更难测

单体应用一个进程搞定，微服务把系统拆成几十上百个服务互相调用，性能测试一下子复杂起来：

```text
单体：
  请求 → [一个大应用] → DB

微服务：
  请求 → 网关 → 订单服务 → 库存服务 → 用户服务 → 支付服务 → ...
                    ↓          ↓          ↓
                  各自的DB   各自的缓存   各自的MQ
```

新增的挑战：

| 挑战 | 说明 |
|------|------|
| **调用链变长** | 一个请求串了 N 个服务，任何一个慢都拖累整体 |
| **瓶颈难定位** | 整体慢，但不知道是哪个服务慢 |
| **木桶效应** | 整条链路的性能由**最慢的那个服务**决定 |
| **依赖复杂** | 服务还依赖网关、注册中心、配置中心、MQ |
| **数据准备难** | 链路上每个服务都要造测试数据 |
| **熔断限流干扰** | 压测可能触发熔断/限流，结果失真 |

---

## 二、三种压测粒度

### 1. 单服务压测（单接口压测）

只压一个服务，看它自己的性能上限。

```text
用途：摸清每个服务的容量基线
做法：直接对目标服务接口发压，把它的依赖 mock 掉或用稳定环境
```

### 2. 链路压测

压一条完整业务链路（如"下单"会经过订单→库存→支付）。

```text
用途：看真实业务路径的端到端性能
难点：要定位链路里哪一环是瓶颈 → 靠分布式追踪
```

### 3. 全链路压测

在**接近生产甚至生产环境**，模拟真实流量压整个系统。

```text
用途：大促前验证整个系统扛不扛得住
难点：流量隔离、数据隔离（不能污染真实数据）
```

> 面试问"微服务怎么压测"：答这三个层次——**先单服务摸基线，再链路找瓶颈，最后全链路验整体**。

---

## 三、微服务特有的瓶颈点

除了前一章的通用瓶颈，微服务还要重点看这些：

### 1. 服务间调用（最关键）

```text
- 同步调用层层嵌套 → 耗时叠加（A等B，B等C...）
- 序列化/反序列化开销（JSON/Protobuf）
- 网络往返(RTT)被放大 N 倍
- 优化：能并行调就并行、能异步就异步、合并调用、加缓存
```

### 2. 网关（流量入口，易成瓶颈）

```text
所有流量都过网关 → 网关一慢全慢
看：网关 CPU、连接数、路由/鉴权/限流逻辑的耗时
```

### 3. 注册中心 / 配置中心

```text
服务发现、配置拉取如果慢或不稳，影响整体
高并发下注册中心压力、心跳风暴
```

### 4. 熔断、限流、降级

```text
压测时如果触发了熔断/限流，TPS 会被人为压住，不是真实瓶颈
要么调高阈值，要么明确这是"限流后的表现"
反过来：也要专门测限流/熔断本身有没有生效（这是微服务必测项）
```

### 5. 连接池(放大版)

```text
每个服务都有自己的线程池、DB 连接池、HTTP 客户端连接池
链路上任何一个池子太小，都会成为木桶最短板
```

## 四、分布式追踪：链路压测的眼睛

链路一长，"整体慢但不知哪慢"是最大痛点。靠**分布式追踪（Tracing）**解决。

```text
原理：给每个请求打一个全局 TraceID，贯穿所有服务
     每个服务记录自己的处理耗时(Span)
     最后能画出一条完整的调用链，看到每一环花了多久
```

```text
一个请求的调用链示例：
网关        ▏ 5ms
└订单服务   ▏▏▏ 30ms
  ├库存服务 ▏▏ 20ms
  └支付服务 ▏▏▏▏▏▏▏▏ 200ms  ← 瓶颈一眼看出来在支付服务！
```

常见工具：**SkyWalking、Jaeger、Zipkin**。

> 配合"可观测性三件套"理解：**Metrics（指标）、Logging（日志）、Tracing（追踪）**。压测分析微服务，Tracing 是定位"哪个服务慢"的利器。

## 五、全链路压测怎么做（进阶了解）

大厂大促前的核心手段，在**生产环境**用真实流量规模压测。两大难题：

### 1. 流量染色（区分压测流量和真实流量）

```text
给压测请求打标记（如 Header 里带压测标）
这个标记随调用链一路透传下去
系统识别出这是压测流量，特殊处理
```

### 2. 数据隔离（不能污染生产数据）

```text
影子库/影子表：压测数据写到专门的"影子"表，不碰真实表
压测流量识别后自动路由到影子库
```

> 这块岗位上手做的机会不多，但**面试能讲清"流量染色 + 影子库"两个概念**就很加分。

## 六、微服务压测的实施步骤

```text
1. 梳理被测链路：画出请求经过哪些服务
2. 单服务摸基线：每个服务单独压，记录各自上限
3. 准备数据和环境：每个服务的测试数据、mock 外部不可控依赖
4. 配好监控和追踪：每个服务的资源监控 + 分布式追踪
5. 链路压测：按业务比例发压，观察端到端表现
6. 定位瓶颈：用 Tracing 找最慢的服务，用监控找资源短板
7. 优化验证：优化木桶最短板 → 再压 → 看整体提升
8. （进阶）全链路压测：流量染色 + 影子库，验证整体容量
```

记住核心思想：**微服务的整体性能 = 最慢一环的性能（木桶效应）。优化要先找最短的那块板。**

---

## 本节重点

- 微服务难在：**调用链长、瓶颈难定位、木桶效应、依赖复杂**
- 三种粒度：**单服务摸基线 → 链路找瓶颈 → 全链路验整体**
- 微服务特有瓶颈：**服务间调用、网关、注册中心、熔断限流、各级连接池**
- **分布式追踪（SkyWalking/Jaeger/Zipkin）** 是定位"哪个服务慢"的核心
- 可观测性三件套：**Metrics、Logging、Tracing**
- 全链路压测两大招：**流量染色 + 影子库**
- 核心思想：整体性能由最慢一环决定，先优化木桶最短板
