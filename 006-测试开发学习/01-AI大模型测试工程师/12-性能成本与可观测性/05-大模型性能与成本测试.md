# 05 大模型性能与成本测试

> 这一节是 AI 大模型测试工程师区别于普通性能测试的地方。大模型接口**又慢、又贵、还按用量计费**，所以性能和成本必须放在一起测。

## 学习目标

学完这一节，你需要能：

- 说清大模型接口性能和传统接口的不同
- 掌握大模型特有的性能指标（首 token 延迟、吐字速度等）
- 用 JMeter 压测大模型接口要注意什么
- 测算和监控大模型调用成本
- 把性能、成本、可观测性结合起来评估

---

## 一、大模型接口为什么特殊

普通接口几十毫秒返回，大模型接口可能要几秒甚至几十秒，而且是**流式**一个字一个字往外吐。

```text
普通接口：    请求 → 等 50ms → 一次性返回完整结果
大模型接口：  请求 → 等 800ms 出第一个字 → 持续吐字 5s → 结束
```

三个根本不同：

```text
1. 慢：响应是秒级，不是毫秒级
2. 流式：结果是流式输出，不是一次性
3. 贵：按 token 计费，调用越多、上下文越长越贵
```

---

## 二、大模型特有的性能指标

普通的 RT/TPS 不够用了，要加这几个：

| 指标 | 含义 | 为什么重要 |
|------|------|-----------|
| **TTFT（首 token 延迟）** | 从发请求到吐出第一个字的时间 | 直接决定用户"等待感"，最关键 |
| **TPOT / 吐字速度** | 每个 token 的间隔 / 每秒输出多少 token | 决定"刷得快不快"的体验 |
| **端到端延迟** | 从请求到完整结果 | 总耗时 |
| **Tokens/s（吞吐）** | 系统每秒能生成多少 token | 衡量服务整体吞吐 |
| **并发能力** | 能同时处理多少路对话 | GPU 资源有限，并发上限低 |

> 面试点："大模型接口性能看什么？" 答：除了 TPS/RT，更要看 **TTFT（首字延迟）和吐字速度**，因为流式场景下用户体验主要由这两个决定。

---

## 三、用 JMeter 压测大模型接口的注意点

大模型接口压测和普通接口有几个坑：

```text
1. 超时要设长：普通接口超时设几秒，大模型要设几十秒，否则全是误报超时
2. 流式响应处理：SSE/流式返回，JMeter 默认可能只取到部分，需特殊处理或测非流式
3. 并发不宜过高：GPU 资源有限，几十并发可能就到上限，别按 Web 接口的几千并发来想
4. 关注 TTFT：普通监听器看的是完整响应时间，首 token 延迟要单独测量
5. 控制成本：压测就是大量真实调用，每次都花钱！要先估算压测本身的费用
6. 输入输出长度要固定：prompt 长度、max_tokens 不同，耗时和费用差很多，要控制变量
```

> 特别提醒：**压测大模型 = 烧钱**。压之前先算一笔账：并发 × 时长 × 每次 token 数 × 单价 = 这次压测要花多少钱。用便宜模型先调通脚本，再上目标模型。

## 四、成本测试与测算

大模型按 token 计费，成本是必测项。

### 成本怎么算

```text
单次成本 = 输入 token 数 × 输入单价 + 输出 token 数 × 输出单价

注意：
- 输入和输出单价通常不同（输出一般更贵）
- 上下文越长、历史越多，输入 token 越多，越贵
- RAG 把检索内容塞进 prompt，会大幅增加输入 token
```

### 成本测试要关注什么

| 关注点 | 说明 |
|--------|------|
| 单请求平均 token 消耗 | 输入/输出各多少 |
| 单请求平均成本 | 折算成钱 |
| 高峰期总成本预估 | 日调用量 × 单次成本 |
| 成本异常 | prompt 是否冗余、有没有重复调用、上下文是否过长 |

### 省成本的常见手段（测试要验证有没有生效）

```text
- 模型分层：简单任务用便宜小模型(haiku)，难任务才用大模型(opus)
- Prompt 缓存：重复的系统提示/上下文走缓存，省输入 token
- 控制 max_tokens：别让模型输出过长
- 精简 prompt：去掉冗余内容
- 结果缓存：相同问题直接返回缓存结果
```

> 测试工程师要能验证这些优化"确实省了钱"——对比优化前后的 token 消耗和成本。

---

## 五、性能 × 成本 × 质量的三角权衡

大模型测试不能只看一个维度，要权衡：

```text
        质量(好不好)
         /\
        /  \
       /    \
性能(快不快)──成本(贵不贵)

大模型 → 质量高，但慢、贵
小模型 → 快、便宜，但质量可能差
```

测试要给出的结论是：**在质量达标的前提下，哪个方案性能和成本最优。** 比如：

> "换成 haiku 后，准确率从 92% 降到 89%（仍达标），但 TTFT 从 1.2s 降到 0.4s，单次成本降低 80%，推荐换用。"

这种带数据的多维结论，才是大模型测试的价值。

---

## 六、可观测性（线上要能看见）

光压测还不够，线上要持续监控。可观测性三件套用到大模型：

| 维度 | 大模型场景具体看什么 |
|------|---------------------|
| **Metrics（指标）** | QPS、TTFT、吐字速度、token 消耗、成本、错误率、限流次数 |
| **Logging（日志）** | 每次调用的 prompt、响应、token 数、耗时（注意脱敏） |
| **Tracing（追踪）** | RAG/Agent 多步调用的链路耗时，定位哪一步慢 |

> 大模型应用常常是"检索 + 多次模型调用 + 工具调用"的长链路，Tracing 能看出是检索慢、还是某次模型调用慢、还是工具慢。

---

## 本节重点

- 大模型接口：**慢、流式、按 token 计费**，和普通接口完全不同
- 特有指标：**TTFT（首字延迟）、吐字速度、Tokens/s**，比单纯 RT 更重要
- 压测注意：**超时设长、并发别太高、控制 token 长度、算清压测花的钱**
- 成本测试：**单次成本 = 输入token×输入价 + 输出token×输出价**，验证省钱优化是否生效
- 三角权衡：在**质量达标前提下**找性能和成本最优解
- 可观测性三件套都要覆盖 token 和成本维度
