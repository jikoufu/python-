# 01 正常测试用例设计方法

## 学习目标

学完这一节，你需要掌握：

- 测试用例是什么。
- 设计测试用例应该从哪些方面考虑。
- 正向、反向、边界、等价类分别怎么用。
- 如何写一条规范的测试用例。
- 如何把测试点整理成测试用例。

## 测试用例是什么

测试用例是为了验证某个功能是否符合预期而设计的一组测试步骤、测试数据和预期结果。

简单说：

```text
我要测什么？
怎么测？
用什么数据测？
期望看到什么结果？
```

一条完整测试用例通常包含：

- 用例编号
- 用例标题
- 前置条件
- 测试步骤
- 测试数据
- 预期结果
- 优先级
- 用例类型

## 测试用例设计的整体思路

设计测试用例时，可以按这个顺序思考：

```text
需求理解
    ↓
正向流程
    ↓
反向流程
    ↓
边界值
    ↓
等价类
    ↓
业务规则
    ↓
状态流转
    ↓
权限控制
    ↓
异常场景
    ↓
兼容性 / 性能 / 安全
```

记住一句话：

```text
正常能不能用，错误能不能拦，边界会不会崩，权限会不会越，异常会不会乱。
```

## 1. 需求理解

设计测试用例前，先看懂需求。

要弄清楚：

- 这个功能是做什么的？
- 谁会使用？
- 输入是什么？
- 输出是什么？
- 业务规则是什么？
- 成功条件是什么？
- 失败条件是什么？
- 是否有权限限制？
- 是否有状态变化？

例如登录功能：

```text
输入：用户名、密码
输出：登录成功 / 登录失败
规则：账号存在、密码正确、账号状态正常才能登录
```

如果需求没看懂，后面的测试用例很容易漏。

## 2. 正向用例

正向用例也叫正常流程用例。

它验证用户按照正确方式操作时，功能是否正常。

登录示例：

```text
正确用户名 + 正确密码 → 登录成功
```

新增用户示例：

```text
填写合法用户名、手机号、邮箱 → 新增成功
```

正向用例一般优先级最高，因为它验证核心业务流程是否可用。

## 3. 反向用例

反向用例验证错误输入或错误操作时，系统是否能正确拦截。

登录示例：

```text
正确用户名 + 错误密码 → 登录失败
不存在的用户名 + 任意密码 → 登录失败
用户名为空 → 提示用户名必填
密码为空 → 提示密码必填
账号被禁用 → 登录失败
```

反向用例的重点不是让系统成功，而是验证系统是否能正确失败。

好的失败表现包括：

- 不崩溃
- 不产生脏数据
- 给出明确提示
- 不泄露敏感信息
- 不绕过业务规则

## 4. 边界值

边界值用来测试输入范围的临界点。

很多 bug 都出现在边界位置。

例如用户名要求 6 到 20 位：

```text
5 位 → 失败
6 位 → 成功
20 位 → 成功
21 位 → 失败
```

年龄要求 18 到 60：

```text
17 → 失败
18 → 成功
60 → 成功
61 → 失败
```

常见边界：

- 最小值
- 最大值
- 最小值 - 1
- 最大值 + 1
- 空值
- 超长值
- 0
- 负数
- 小数

## 5. 等价类

等价类是把大量输入分成几类，每类选一个代表值测试。

这样可以减少测试数量，同时保持覆盖率。

例如年龄要求 18 到 60：

```text
有效等价类：18 到 60
无效等价类：小于 18
无效等价类：大于 60
无效等价类：非数字
无效等价类：为空
```

测试数据可以选：

```text
25 → 有效
17 → 无效
61 → 无效
abc → 无效
空 → 无效
```

等价类适合字段校验、表单输入、接口参数测试。

## 6. 业务规则

业务规则是测试用例设计的重点。

很多问题不是字段格式错，而是业务规则没处理好。

优惠券示例：

```text
满 100 可用
未满 100 不可用
过期不可用
已使用不可用
指定商品可用
不同优惠券不能叠加
```

订单示例：

```text
待支付订单可以取消
已支付订单不能直接删除
已发货订单不能修改地址
已完成订单可以申请售后
```

业务规则要尽量从需求文档、产品说明、原型、接口文档里提取。

## 7. 状态流转

如果功能有状态，就必须测试状态流转。

订单状态示例：

```text
待支付 → 已支付
已支付 → 已发货
已发货 → 已完成
待支付 → 已取消
```

还要测试不允许的状态变化：

```text
已完成 → 待支付，不允许
已取消 → 已支付，不允许
已发货 → 修改订单金额，不允许
```

适合状态流转测试的功能：

- 订单
- 工单
- 审批流
- 支付
- 退款
- 任务执行
- Agent 多步骤任务

## 8. 权限控制

权限测试验证不同角色能不能访问或操作对应资源。

常见角色：

```text
未登录用户
普通用户
管理员
超级管理员
只读用户
企业用户
个人用户
```

订单权限示例：

```text
普通用户只能查看自己的订单
普通用户不能查看别人的订单
管理员可以查看所有订单
未登录用户不能访问订单
```

权限测试要重点关注：

- 未登录访问
- Token 过期
- 普通用户越权
- 横向越权
- 纵向越权
- 管理员接口保护

## 9. 异常场景

异常场景测试系统在出错时是否稳定。

常见异常：

```text
网络超时
数据库异常
第三方接口失败
重复提交
并发请求
Token 过期
参数格式错误
文件上传失败
磁盘空间不足
服务重启
```

异常场景的预期不是“一定成功”，而是：

- 错误提示清楚
- 系统不崩溃
- 数据不乱
- 事务能回滚
- 日志能定位问题
- 用户不会重复扣费或重复下单

## 10. 兼容性和易用性

如果是 Web 页面，还要考虑：

- Chrome / Edge / Firefox
- 不同屏幕尺寸
- 移动端适配
- 按钮是否可点击
- 输入框提示是否明确
- 错误提示是否清楚
- 加载状态是否合理
- 重复点击是否防抖

如果是接口，则关注：

- 参数兼容
- 旧版本客户端兼容
- 字段新增是否影响旧调用方
- 错误码是否稳定

## 11. 性能和安全

基础功能测完后，还要考虑性能和安全。

性能测试点：

```text
响应时间
并发访问
大数据量
分页性能
接口超时
重复请求
```

安全测试点：

```text
SQL 注入
XSS
越权访问
敏感信息泄露
弱密码
接口未鉴权
文件上传风险
```

AI 大模型应用还要关注：

```text
Prompt 注入
越狱
敏感信息泄露
RAG 数据污染
Agent 越权调用工具
模型输出有害内容
```

## 测试用例模板

```markdown
## 用例编号

TC_001

## 用例标题

正确账号密码登录成功

## 前置条件

- 用户账号已存在。
- 用户账号状态正常。

## 测试步骤

1. 打开登录页。
2. 输入正确用户名。
3. 输入正确密码。
4. 点击登录按钮。

## 测试数据

用户名：admin
密码：123456

## 预期结果

- 登录成功。
- 跳转到首页。
- 页面显示当前用户信息。
- 后端返回登录 token。

## 优先级

高

## 用例类型

正向用例
```

## 登录功能测试点示例

| 类型 | 测试点 | 预期结果 |
|------|--------|----------|
| 正向 | 正确用户名和密码 | 登录成功 |
| 反向 | 密码错误 | 登录失败 |
| 反向 | 用户名不存在 | 登录失败 |
| 必填 | 用户名为空 | 提示用户名必填 |
| 必填 | 密码为空 | 提示密码必填 |
| 边界 | 密码长度小于最小限制 | 提示格式错误 |
| 权限 | 被禁用账号登录 | 登录失败 |
| 安全 | 多次输错密码 | 触发限制或验证码 |
| 异常 | 登录接口超时 | 给出友好提示 |

## 接口测试用例示例

```markdown
## 用例编号

API_USER_001

## 用例标题

根据用户 ID 查询用户成功

## 请求方法

GET

## 请求地址

/users/1

## 请求参数

user_id = 1

## 前置条件

数据库中存在 ID 为 1 的用户。

## 预期结果

- 状态码为 200。
- 返回 JSON 中 `id` 等于 1。
- 返回 JSON 中包含 `username`。
- 不返回 `password` 字段。

## 优先级

高
```

## AI 应用测试用例示例

```markdown
## 用例编号

AI_RAG_001

## 用例标题

知识库中存在答案时，RAG 能正确回答

## 前置条件

- 知识库已导入 FastAPI 学习笔记。
- 笔记中包含 Depends 的说明。

## 用户问题

FastAPI 的 Depends 是什么？

## 预期结果

- 回答不为空。
- 回答包含“依赖注入”。
- 回答基于知识库内容。
- 不编造知识库中没有的信息。
- 如果系统支持引用，应返回来源文档。

## 优先级

高
```

## 常见漏测点

设计测试用例时，容易漏这些：

- 空值
- 超长输入
- 特殊字符
- 重复提交
- 并发操作
- 权限越权
- 状态不可逆
- 旧数据兼容
- 第三方接口失败
- 异常提示
- 日志记录
- 敏感信息泄露

## 本节总结

测试用例设计不是随便列几个正常操作，而是要系统性覆盖：

- 正常流程
- 错误流程
- 边界值
- 等价类
- 业务规则
- 状态流转
- 权限控制
- 异常场景
- 兼容性
- 性能和安全

最终目标是：

```text
用尽量少但有效的用例，覆盖最重要的风险。
```
